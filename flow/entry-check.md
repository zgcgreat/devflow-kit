# 入场检测详细规则

> 本文件是 GO.md 第二步的详细展开，由 GO.md 按需加载。

## 强制门控

入场检测是**进入任何需求相关阶段的前置门**。未完成本步检测前，AI **禁止**进入以下任一阶段：
- 0-confirm（需求确认）
- 1-analysis（需求分析）
- 2-design（方案设计）
- 文档解析模式（D.1/D.2/D.3）

**例外**：横向命令（A-architect / A-evolve / I-intel-scan / L-restyle / M-health）不跑本步。

---

## 决策树（按顺序判定，命中即停止）

```
1. 项目状态.md 有 ai_context_doc 字段？
   └─ 是 → 情况 A（直接用指定文档）
   └─ 否 → 继续

2. .specs/上下文.md 存在？
   ├─ 存在且 last_intel_scan < 90天 → 情况 B（直接用）
   ├─ 存在且 last_intel_scan > 90天 → 情况 C（提醒可重扫）
   └─ 不存在 → 继续

3. 检测到其他 AI 上下文文档？
   └─ 是 → 情况 D（停等用户确认用哪个）
   └─ 否 → 继续

4. 项目是 greenfield？
   └─ 是 → 情况 F（跳过）
   └─ 否 → 情况 E（停等用户确认）
```

## 情况详解

### 情况 A · 项目状态.md 已设 ai_context_doc

**判定条件**：`.specs/项目状态.md` 文件存在 **且** 其中有 `ai_context_doc: <路径>` 字段（非空、非 none）。

**动作**：直接读用户指定的文档，不再询问。

**检测结果框**：
```
┌─────────────────────────────────────────────────────────────┐
│  🔍 入场检测                                                  │
├─────────────────────────────────────────────────────────────┤
│  检测结果：情况 A                                             │
│  项目类型：brownfield                                         │
│  上下文文档：<用户指定的文档路径>                              │
│  入场扫描状态：已完成                                          │
└─────────────────────────────────────────────────────────────┘
```

**更新项目状态.md**：
- `项目类型`: brownfield
- `入场扫描状态`: 已完成
- `上下文文档`: <用户指定的文档路径>
- `跳过原因`: 用户指定现有文档

---

### 情况 B · 已存在 上下文.md 且在 90 天内

**判定条件**：`.specs/上下文.md` 文件存在 **且** 项目状态.md 的 `last_intel_scan` 字段在 90 天内（或字段为空但上下文.md 存在）。

**动作**：直接读 上下文.md，跳过本步，无需打扰用户。

**检测结果框**：
```
┌─────────────────────────────────────────────────────────────┐
│  🔍 入场检测                                                  │
├─────────────────────────────────────────────────────────────┤
│  检测结果：情况 B                                             │
│  项目类型：brownfield                                         │
│  上下文文档：上下文.md                                         │
│  入场扫描状态：已完成                                          │
└─────────────────────────────────────────────────────────────┘
```

---

### 情况 C · 已存在 上下文.md 但超过 90 天

**判定条件**：`.specs/上下文.md` 文件存在 **且** 项目状态.md 的 `last_intel_scan` 字段超过 90 天。

**动作**：读 上下文.md，**提醒用户**可重跑 intel-scan，但**不强制**。

**检测结果框**：
```
┌─────────────────────────────────────────────────────────────┐
│  🔍 入场检测                                                  │
├─────────────────────────────────────────────────────────────┤
│  检测结果：情况 C                                             │
│  项目类型：brownfield                                         │
│  上下文文档：上下文.md                                         │
│  入场扫描状态：已完成（上次扫描超过 90 天，建议重扫）          │
└─────────────────────────────────────────────────────────────┘
```

---

### 情况 D · 检测到其他 AI 上下文文档

**判定条件**：`.specs/上下文.md` 不存在 **且** 项目状态.md 无 `ai_context_doc` 字段 **且** 检测到以下任一文件：
- `AGENTS.md`（仓库根）
- `CLAUDE.md`（仓库根 / `.claude/`）
- `.cursor/rules/*.md`
- `.windsurf/rules/*.md`
- `.github/copilot-instructions.md`
- `.clinerules`

**动作**：**必须停下来等用户确认**。

**检测结果框**：
```
┌─────────────────────────────────────────────────────────────┐
│  🔍 入场检测                                                  │
├─────────────────────────────────────────────────────────────┤
│  检测结果：情况 D                                             │
│  项目类型：brownfield                                         │
│  上下文文档：发现 <N> 个候选                                   │
│  入场扫描状态：待用户确认                                      │
└─────────────────────────────────────────────────────────────┘
```

**反问用户**（必须等待回复）：
```
🔍 检测到本项目已有以下 AI 上下文文档：
  - <列出找到的，含路径与文件大小>

devflow-kit 默认用 上下文.md 作为单一源。请选择：
  1. 跑 intel-scan，综合现有文档 + 代码扫描，生成 上下文.md（推荐）
  2. 以现有文档为准（告诉我哪个），跳过 intel-scan
  3. 跳过 intel-scan，直接进 0-confirm（不推荐 · AI 会"盲飞"）

请选 1/2/3。
```

### 情况 F · Greenfield 项目

**判定条件**：项目根目录不存在以下任一文件：
- `package.json`（Node/JS）
- `pyproject.toml` / `requirements.txt`（Python）
- `go.mod`（Go）
- `Cargo.toml`（Rust）
- `pom.xml` / `build.gradle`（Java/Kotlin）
- `composer.json`（PHP）
- `Gemfile`（Ruby）

**动作**：跳过本步。这是 greenfield 项目，上下文.md 会在后续阶段逐步沉淀。

## 为什么这步重要

跳过会导致：
- AI 不知项目架构 → 写出不合项目风格的代码
- AI 重复实现已有抽象 → 费 token 且产生重复代码
- 4-dev 的 schema 任务 / 破坏性变更检测都会不准
- 用户明明有 CLAUDE.md 写好的约定 → AI 完全不读 → 抱怨"为什么不按我说的来"
