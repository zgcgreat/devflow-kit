# 模式判定与切换规则

> 本文件是 GO.md 模式部分的详细展开，由 GO.md 按需加载。

## 三档模式概览

| 模式 | 适用 | 完整流程 | 产物清单 |
|------|------|----------|----------|
| **Fast** | 1~2 文件、低风险、通常 <50 行、需求清楚、可快速回滚 | 短任务说明 → 修改 → 窜验证 → 结果证据 | `开发记录.md`（可选） |
| **Standard** | 常规 feature / bugfix、多文件、UI、业务逻辑、需要 AC | 需求确认 → 需求分析 → 方案设计 → [2a UI设计]* → 任务拆分 → 开发 → 测试 → 审查 → 集成 | `00-需求确认.md` → `01-需求分析.md` → `02-方案设计.md` → [`02a-UI设计.md`] → `03-任务拆分.md` → `*-开发记录.md` → `05-测试报告.md` → `06-代码审查.md` |
| **Strict** | 高风险 / 生产敏感 / 架构或数据影响 | 完整流程 + 明确确认点 + 安全/迁移/回滚/独立 review | Standard 全部产物 + 安全审查报告 + 迁移方案 + 回滚方案 |

> *2a UI设计 仅前端项目需要

---

## 模式确认规则（风险驱动）

| 判定结果 | 是否停等用户 | 规则 |
|----------|--------------|------|
| **Fast** | 默认不必停等 | 在路由声明中说明 Fast 理由后可直接执行；**用户反对或发现隐藏风险时必须切换到 Standard/Strict 并停等确认**。 |
| **Standard** | 建议停等 | 新功能、多文件、UI 或需要 AC 的工作，先给模式建议并等用户确认；用户明确要求"直接做"时可继续但记录裁剪。 |
| **Strict** | 必须停等 | 命中高风险矩阵时必须单独确认模式、范围、风险与确认门，用户确认后才能继续。 |

## Fast 模式切换规则

### Fast 模式准入 Checklist（v2.0 新增）

**进入 Fast 模式前，AI 必须确认以下所有条件：**

#### 硬性指标（必须全部满足，任一不满足 → 强制 Standard）

- [ ] 改动文件数 ≤ 2
- [ ] 预估代码行数 < 50（不含空行/注释）
- [ ] 不涉及数据库 schema 变更
- [ ] 不涉及 API 接口签名变更
- [ ] 不涉及鉴权/权限/支付逻辑

#### 软性指标（至少满足 3/5 项，否则建议 Standard）

- [ ] 需求描述清晰无歧义
- [ ] 有明确的验收标准
- [ ] 不影响其他模块（非公共组件）
- [ ] 有现成的测试框架可直接运行
- [ ] 非核心业务逻辑

#### 自动排除项（命中任一项 → 强制 Standard）

❌ 修改订单/支付/用户认证核心逻辑  
❌ 修改公共组件/工具函数（被多处引用）  
❌ 涉及第三方依赖版本升级  
❌ 需要修改 CI/CD 配置  
❌ 需要数据迁移或回滚方案  

### Fast 模式检查输出格式

AI 在判定 Fast 模式时，必须输出检查结果：

```
🔍 Fast 模式检查

硬性指标:
✅ 改动文件数: 1个
✅ 预估行数: 30行
✅ 无schema变更
✅ 无API变更
✅ 非鉴权逻辑

软性指标:
✅ 需求清晰
✅ 有验收标准
❌ 影响其他模块（搜索组件被多处引用）
✅ 有测试框架
✅ 非核心逻辑

结果: 5/5硬性通过，4/5软性通过
判定: ✅ 符合 Fast 模式
```

如软性指标 < 3/5，输出警告：

```
⚠️ Fast 模式检查警告

硬性指标: 5/5通过
软性指标: 2/5通过（低于阈值3/5）

建议: 升级到 Standard 模式

原因:
- 影响其他模块（搜索组件被多处引用）
- 无明确验收标准

请确认:
1. 继续 Fast 模式（我了解风险）
2. 升级到 Standard（推荐）
```

---

### Fast → Standard/Strict 切换规则

Fast → Standard/Strict：如果用户反对或 AI 发现隐藏风险（如改动超预期、涉及 schema 等），**必须暂停并输出模式切换确认**：

```
⚠️ Fast 模式切换为 Standard
   触发原因：<具体原因，如"发现涉及 schema 变更">
   新流程：<Standard 完整流程>

   是否同意切换？[Y/n]
```

- 用户确认后才可切换模式，AI 不得自行切换。

## Standard / Strict 模式确认格式

```
🎯 建议模式：<Standard / Strict>
   理由：<简要说明为什么是这个模式>
   流程：<该模式对应的最小流程，展开写>

   其他可选模式：
   • Fast — 短任务说明 → 修改 → 窜验证 → 结果证据（仅低风险小改动）
   • Strict — 完整流程 + 安全/迁移/回滚/独立 review（高风险改动）

   请确认或选择其他模式：
   1. ✅ <建议模式>（推荐）
   2. Fast
   3. Strict
```

**示例**：

```
🎯 建议模式：Standard
   理由：新增搜索功能，涉及前后端多文件改动，有 UI 变化

   📋 Standard 完整流程：
   ┌─────────────────────────────────────────────────────────────┐
   │  需求确认 → 需求分析 → 方案设计 → [2a UI设计] → 任务拆分    │
   │      ↓                                                       │
   │  开发 → 测试 → 审查 → 集成                                    │
   └─────────────────────────────────────────────────────────────┘
   产物：00-需求确认.md → 01-需求分析.md → 02-方案设计.md
        → [02a-UI设计.md] → 03-任务拆分.md → *-开发记录.md
        → 05-测试报告.md → 06-代码审查.md

   其他可选模式：
   ┌─────────────────────────────────────────────────────────────┐
   │ Fast — ≤2 文件、<50 行、低风险                               │
   │ 流程：短任务说明 → 修改 → 窜验证 → 结果证据                   │
   │ 产物：开发记录.md（可选）                                     │
   ├─────────────────────────────────────────────────────────────┤
   │ Strict — 高风险/生产敏感/架构或数据影响                       │
   │ 流程：完整流程 + 安全审查 + 迁移方案 + 回滚方案               │
   │ 产物：Standard 全部 + 安全审查报告 + 迁移方案 + 回滚方案      │
   └─────────────────────────────────────────────────────────────┘

   请确认或选择其他模式：
   1. ✅ Standard（推荐）
   2. Fast
   3. Strict
```

## 模式降级评估（阶段切换时评估，需用户确认）

AI 在完成任一阶段后进入下一阶段前，依据以下信号评估是否应建议降级：

| 信号 | 当前模式 | 可降级到 |
|------|----------|----------|
| 实际改动文件数 ≤ 2，且均为同一层（纯后端/纯前端） | Standard | Fast |
| 所有改动加起来 < 50 行（不含空行/注释/已有代码搬运） | Standard | Fast |
| 无数据库/API/跨模块影响 | Standard | Fast |
| 命中 0-confirm 阶段范围排除的任一条 | Standard | Fast |
| 当前阶段运行中用户表示"太多了/太复杂了" | **任何模式** | 降一级或暂停 |

降级前 AI 必须暂停并询问用户确认：

```
📉 建议模式降级：Standard → Fast
   触发信号：实际改动仅 2 文件 38 行，纯后端无 schema 变更
   裁剪内容：跳过 2-design / 2a-ui-design，测试报告合并到开发记录
   是否同意降级？[Y/n]
```

用户回复确认后才执行降级；用户拒绝则保持当前模式继续。

不允许「悄无声息地跳过阶段」，也不允许 AI 自行决定降级。

## 高风险自动升级 Strict

命中以下任一关键词自动升级到 Strict：

- 鉴权、权限、支付、账单
- secret、PII、数据保留
- schema / migration
- 公共 API / SDK
- infra、部署、CI/CD、环境配置
- 依赖升级、跨模块架构
- 并发、后台任务、缓存
- 数据丢失风险、不可逆操作
- 上线回滚敏感路径

---

## 风险评估矩阵（v2.3 新增）

### 评估维度

| 维度 | 低 (1分) | 中 (2分) | 高 (3分) |
|------|---------|---------|----------|
| **技术复杂度** | 简单CRUD、单文件修改 | 多文件协调、需理解现有架构 | 新算法/协议、分布式系统、性能优化 |
| **业务影响** | 内部工具、非核心功能 | 用户-facing功能、常规业务流程 | 支付/订单/鉴权、核心营收链路 |
| **数据敏感性** | 无持久化、临时数据 | 普通业务数据 | PII、金融数据、医疗数据、密钥 |
| **回滚难度** | 代码回退即可、无数据变更 | 需数据迁移但可逆 | 不可逆操作、需人工介入恢复 |

### 评分规则

```python
total_score = sum(各维度得分)
max_score = 12 (4维度 × 3分)

if any(维度 == "高") or total_score >= 8:
    mode = "Strict"
elif total_score >= 5:
    mode = "Standard"
else:
    mode = "Fast"
```

### 评估输出格式

AI 在模式判定时必须输出风险评估：

```markdown
🎯 模式推荐：Strict（置信度 90%）

📊 风险评估矩阵:
┌──────────────┬───────┬────────────────────────┐
│ 维度         │ 等级  │ 理由                   │
├──────────────┼───────┼────────────────────────┤
│ 技术复杂度   │ 中(2) │ 需理解现有Auth中间件     │
│ 业务影响     │ 高(3) │ 涉及用户登录核心流程     │
│ 数据敏感性   │ 中(2) │ JWT token，非PII        │
│ 回滚难度     │ 低(1) │ 代码回退即可            │
└──────────────┴───────┴────────────────────────┘
总分: 8/12 → Strict模式 (≥8分)

⚠️ 自动升级因素:
- 总分 ≥8 → 强制 Strict
- 业务影响为"高" → 建议加强测试覆盖

评分详情:
- Fast: 10%
- Standard: 30%
- Strict: 60%

请确认或选择其他模式：
1. ✅ Strict（推荐）
2. Standard (降级需确认风险)
3. Fast
```

**注意**: 如果用户认为 Strict 过于严格,可以确认降级到 Standard,但 AI 必须记录风险提示。

### 特殊规则

1. **一票否决**: 任一维度为"高"且涉及以下场景 → 强制 Strict
   - 支付/金融交易
   - 用户认证/授权
   - 数据库schema变更
   - 生产环境部署

2. **保守策略**: 当置信度 < 60% 时 → 建议升级到更高级模式

3. **用户覆盖**: 用户可显式指定模式覆盖AI推荐，但AI必须记录风险提示

## 必须人工确认的门

| 事项 | 说明 |
|------|------|
| 需求范围最终确认 | 00-需求确认.md 签字 |
| 技术栈或架构方向锁定 | 02-方案设计.md ## 0 段 |
| 新依赖、schema / migration、删除文件 | 显式确认 |
| CI/CD、部署、生产发布、不可逆操作 | Strict 模式必须确认 |
| 接受 Critical review 发现而不修复 | 显式确认 |
| 模式升级 / 降级 | 用户确认后执行 |

## Fast 路由声明要求

Fast 路由声明必须包含：为什么可 Fast、触碰范围、验证方式。

```
✅ 模式：Fast
✅ 改动范围：<文件列表>
✅ verify 命令：<具体命令>
✅ 预期结果：<通过标准>
```

---

## 例外情况

用户在需求中已显式指定模式（如"Fast 模式：修个 typo"）→ 可直接采用，但仍要在路由声明中展示该模式和流程（不再列出其他选项），让用户有机会确认。
