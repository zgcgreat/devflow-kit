# devflow-kit Stage: 5-Test（测试验证）

> **阶段定位**：验证所有AC是否通过
> **前置条件**：4-dev阶段所有任务完成
> **后置产物**：`.devflow-kit/<req-id>/05-测试报告.md`

## Skill元信息

```yaml
name: stage-5-test
version: 1.0.0
description: devflow-kit工作流第5阶段 - 测试验证与报告生成
author: devflow-kit
dependencies:
  - testing-suite
```

## 输入

- `.devflow-kit/<req-id>/01-需求分析.md`（或Delta版）
- `.devflow-kit/<req-id>/02-方案设计.md`（必读 `## 0. 技术栈选定`）

- `.devflow-kit/<req-id>/04-开发记录.md` + 各 `<task-id>-开发记录.md`

- `.devflow-kit/<req-id>/05-测试报告.md`

#### 1.2 分级读取策略

**🔴 必须读取**（缺失会阻塞流程）：
- **01-需求分析.md**：提供AC列表，是测试的核心依据
  - 如果缺失 → 报错并引导回stage-1-analysis

**🟡 建议读取**（缺失采用降级策略）：
- **00-需求确认.md**：提供模式判定，决定测试轮数
  - 如果缺失 → 默认使用Standard模式（4轮测试）
- **02-方案设计.md**：提供技术栈，决定测试命令
  - 如果缺失 → 从package.json推断测试框架
- **03-任务拆分.md**：提供任务清单，决定测试范围
  - 如果缺失 → 从04-开发记录推断改动范围
- **04-开发记录.md**：提供关键决策，帮助理解实现细节
  - 如果缺失 → 直接测试，但需人工验证更多场景

**🟢 可选读取**（补充信息）：
- **上下文.md**：提供编码规范
  - 如果缺失 → 使用通用命名约定
- **经验总结.md**：提供历史测试教训
  - 如果缺失 → 使用通用测试策略

### Step 2: 确定测试策略

根据项目类型和技术栈，参考 `test-pyramid.md` 的「适用矩阵」：

| 测试类型 | 占比 | 说明 |
|----------|------|------|
| 单元测试 | 70% | 函数/组件级别 |
| 集成测试 | 20% | 模块间交互 |
| E2E测试 | 10% | 完整用户流程 |

**按节读取test-pyramid.md**，不要整读。

### Step 3: 第一轮 - 单元测试验证

**目标**：验证每个任务的单元测试通过

**执行**：
```bash
# 运行所有单元测试
npm test
# 或
pytest
# 或
go test ./...
```

**记录结果**：
```markdown
## 第一轮：单元测试

**命令**: `npm test`

**结果**:
- Test Suites: 15 passed, 15 total
- Tests:       87 passed, 87 total
- Coverage:    85% statements, 82% branches

**失败用例**（如有）:
- ❌ test_xxx: <错误信息>
  - 原因: ...
  - 修复: ...
```

**要求**：
- 覆盖率 ≥ 80%（Standard模式）
- 覆盖率 ≥ 90%（Strict模式）
- 无失败用例

### Step 4: 第二轮 - 集成测试验证

**目标**：验证模块间交互正确

**执行**：
```bash
# 运行集成测试
npm run test:integration
# 或
pytest tests/integration/
```

**重点验证**：
- API接口调用链
- 数据库事务
- 缓存一致性
- 消息队列处理

**记录结果**：
```markdown
## 第二轮：集成测试

**命令**: `npm run test:integration`

**结果**:
- 测试用例数: 12
- 通过: 12
- 失败: 0

**关键场景**:
- ✅ 通知创建 → 写入DB → 发布MQ → 发送邮件
- ✅ 缓存命中 → 不查DB
- ✅ 缓存未命中 → 查DB → 写缓存
```

### Step 5: 第三轮 - AC覆盖验证

**目标**：逐一验证每个AC是否通过

**方法**：
1. 对每个AC，编写/执行对应的测试
2. 手动测试（如UI交互）
3. API测试（如后端接口）

**记录格式**：
```markdown
## 第三轮：AC覆盖验证

| AC ID | 描述 | 测试方法 | 结果 | 备注 |
|-------|------|----------|------|------|
| AC-1 | 用户能发送通知 | 单元测试 + API测试 | ✅ 通过 | - |
| AC-2 | 通知出现在列表 | E2E测试 | ✅ 通过 | - |
| AC-3 | 未读标记正确 | 集成测试 | ✅ 通过 | - |
| AC-4 | P95 < 200ms | 性能测试 | ⚠️ 210ms | 需优化查询 |
```

**要求**：
- 所有AC必须100%覆盖
- Standard模式：允许1个AC有警告（⚠️）
- Strict模式：所有AC必须✅通过

### Step 6: 第四轮 - 回归测试

**目标**：确保新代码未破坏现有功能

**执行**：
```bash
# 运行全量测试套件
npm run test:all
# 或
pytest tests/
```

**重点检查**：
- 核心业务流程
- 公共组件/工具函数
- 数据库Schema兼容性

**记录结果**：
```markdown
## 第四轮：回归测试

**命令**: `npm run test:all`

**结果**:
- 总测试数: 250
- 通过: 248
- 失败: 2
- 退化: 0

**失败分析**:
- ❌ test_legacy_feature: 与本次改动无关，历史问题
- ❌ test_edge_case: 边界条件未处理，已修复
```

### Step 7: 第五轮 - 性能测试（如需要）

**触发条件**：
- AC中有性能要求
- Strict模式
- 用户明确要求

**执行**：
```bash
# 负载测试
npm run perf:load
# 或
ab -n 1000 -c 10 http://localhost:3000/api/notifications
```

**记录指标**：
```markdown
## 第五轮：性能测试

**场景**: 并发10用户，请求1000次

**指标**:
- P50: 85ms
- P95: 180ms ✅ (< 200ms要求)
- P99: 250ms
- QPS: 120
- 错误率: 0.1%

**瓶颈分析**:
- 数据库查询占60%时间
- 建议: 增加索引、优化查询
```

### Step 7.5: 完整性自检

**检查是否充分利用了前置产物**：

- [ ] 是否读取了所有存在的必需产物（01-需求分析）？
- [ ] 是否从01-需求分析中提取了所有AC列表？
- [ ] 是否从00-需求确认中获取了模式判定（如存在）？
- [ ] 是否参考了02-方案设计中的技术栈（如存在）？
- [ ] 是否参考了04-开发记录中的关键决策（如存在）？
- [ ] 对于缺失的产物，是否采用了合理的降级策略？
- [ ] 测试轮数是否符合模式判定？
  - Fast：可能只需单元测试 + AC覆盖验证
  - Standard：4轮测试（单元/集成/AC/回归）
  - Strict：5轮测试（+性能测试）
- [ ] **所有AC都有对应的测试结果**（AC覆盖检查）？

**如果发现遗漏**：
→ 回到Step 1重新读取
→ 或在本阶段产物中注明"因缺少XX产物，采用YY假设"

### Step 8: 读取模板并提取段落清单

**⚠️ 强制规则**：必须使用 `read_file` 工具读取模板文件。

```python

```

**从模板中提取必填段落清单**：
```
□ 测试概览（总轮数、通过率）
□ 每轮测试详细结果
□ AC覆盖矩阵
□ 性能指标（如执行）
□ 问题清单及修复状态
□ 测试结论（通过/不通过）
```

**注意**：
- 所有6个段落都必须包含，不得省略
- 测试结果必须有量化数据，不能只有“通过”
- 测试结论必须明确（通过/有条件通过/不通过）

### Step 9: 生成测试报告并逐项核对

按模板生成 `.devflow-kit/<req-id>/05-测试报告.md`：

- **所有 `<...>` 占位符必须替换为实际值**
- **AC覆盖率必须100%**

**生成后核对**：
```markdown
产物核对清单：
- [ ] 测试概览 → 已包含（总轮数、通过率）
- [ ] 每轮测试详细结果 → 已包含（单元/集成/AC覆盖）
- [ ] AC覆盖矩阵 → 已包含（100%覆盖）
- [ ] 性能指标 → 已包含（Strict模式必填）
- [ ] 问题清单及修复状态 → 已包含
- [ ] 测试结论 → 已包含（通过/有条件通过/不通过）
```

**如果有缺失**：立即补齐，不得进入下一步。

**测试结论判定**：
- ✅ **通过**：所有AC通过，无🔴问题
- ⚠️ **有条件通过**：所有AC通过，有🟡问题但不阻塞上线
- ❌ **不通过**：存在AC未通过或有🔴问题

## 错误处理

- 测试失败 → 分析原因，分类为：代码bug/测试用例错误/环境问题
- 覆盖率不达标 → 补充缺失的测试用例
- AC无法验证 → 回到1-analysis澄清AC
- 性能不达标 → 优化代码或调整AC要求
