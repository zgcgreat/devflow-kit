## 执行流程

### Step 1: 读取AC列表

从 01-需求分析.md 提取所有AC：
- AC-1, AC-2, ..., AC-N
- 每个AC的Given/When/Then

### Step 2: 确定测试策略

根据项目类型和技术栈，参考 `test-pyramid.md` 的「适用矩阵」：

| 测试类型 | 占比 | 说明 |
|----------|------|------|
| 单元测试 | 70% | 函数/组件级别 |
| 集成测试 | 20% | 模块间交互 |
| E2E测试 | 10% | 完整用户流程 |

**按节读取test-pyramid.md**，不要整读。

### Step 3: 第一轮 - 单元测试验证

**目标**：验证每个任务的单元测试通过

**执行**：
```bash
# 运行所有单元测试
npm test
# 或
pytest
# 或
go test ./...
```

**记录结果**：
```markdown
## 第一轮：单元测试

**命令**: `npm test`

**结果**:
- Test Suites: 15 passed, 15 total
- Tests:       87 passed, 87 total
- Coverage:    85% statements, 82% branches

**失败用例**（如有）:
- ❌ test_xxx: <错误信息>
  - 原因: ...
  - 修复: ...
```

**要求**：
- 覆盖率 ≥ 80%（Standard模式）
- 覆盖率 ≥ 90%（Strict模式）
- 无失败用例

### Step 4: 第二轮 - 集成测试验证

**目标**：验证模块间交互正确

**执行**：
```bash
# 运行集成测试
npm run test:integration
# 或
pytest tests/integration/
```

**重点验证**：
- API接口调用链
- 数据库事务
- 缓存一致性
- 消息队列处理

**记录结果**：
```markdown
## 第二轮：集成测试

**命令**: `npm run test:integration`

**结果**:
- 测试用例数: 12
- 通过: 12
- 失败: 0

**关键场景**:
- ✅ 通知创建 → 写入DB → 发布MQ → 发送邮件
- ✅ 缓存命中 → 不查DB
- ✅ 缓存未命中 → 查DB → 写缓存
```

### Step 5: 第三轮 - AC覆盖验证

**目标**：逐一验证每个AC是否通过

**方法**：
1. 对每个AC，编写/执行对应的测试
2. 手动测试（如UI交互）
3. API测试（如后端接口）

**记录格式**：
```markdown
## 第三轮：AC覆盖验证

| AC ID | 描述 | 测试方法 | 结果 | 备注 |
|-------|------|----------|------|------|
| AC-1 | 用户能发送通知 | 单元测试 + API测试 | ✅ 通过 | - |
| AC-2 | 通知出现在列表 | E2E测试 | ✅ 通过 | - |
| AC-3 | 未读标记正确 | 集成测试 | ✅ 通过 | - |
| AC-4 | P95 < 200ms | 性能测试 | ⚠️ 210ms | 需优化查询 |
```

**要求**：
- 所有AC必须100%覆盖
- Standard模式：允许1个AC有警告（⚠️）
- Strict模式：所有AC必须✅通过

### Step 6: 第四轮 - 回归测试

**目标**：确保新代码未破坏现有功能

**执行**：
```bash
# 运行全量测试套件
npm run test:all
# 或
pytest tests/
```

**重点检查**：
- 核心业务流程
- 公共组件/工具函数
- 数据库Schema兼容性

**记录结果**：
```markdown
## 第四轮：回归测试

**命令**: `npm run test:all`

**结果**:
- 总测试数: 250
- 通过: 248
- 失败: 2
- 退化: 0

**失败分析**:
- ❌ test_legacy_feature: 与本次改动无关，历史问题
- ❌ test_edge_case: 边界条件未处理，已修复
```

### Step 7: 第五轮 - 性能测试（如需要）

**触发条件**：
- AC中有性能要求
- Strict模式
- 用户明确要求

**执行**：
```bash
# 负载测试
npm run perf:load
# 或
ab -n 1000 -c 10 http://localhost:3000/api/notifications
```

**记录指标**：
```markdown
## 第五轮：性能测试

**场景**: 并发10用户，请求1000次

**指标**:
- P50: 85ms
- P95: 180ms ✅ (< 200ms要求)
- P99: 250ms
- QPS: 120
- 错误率: 0.1%

**瓶颈分析**:
- 数据库查询占60%时间
- 建议: 增加索引、优化查询
```

### Step 8: 生成测试报告

**⚠️ 强制规则**：输出前必须先读取模板文件 `flow/templates/05-测试报告.md`。

按模板生成 `.specs/<req-id>/05-测试报告.md`，包含：
- **必须包含模板所有段落**（不得省略或改写）
- **所有 `<...>` 占位符必须替换为实际值**
- 测试概览（总轮数、通过率）
- 每轮测试详细结果
- AC覆盖矩阵
- 性能指标（如执行）
- 问题清单及修复状态
- 测试结论（通过/不通过）

**自检**：输出前逐项核对模板强制规则（R13.9 / R13.10）

**测试结论判定**：
- ✅ **通过**：所有AC通过，无🔴问题
- ⚠️ **有条件通过**：所有AC通过，有🟡问题但不阻塞上线
- ❌ **不通过**：存在AC未通过或有🔴问题

### Step 9: 更新项目状态

```markdown
当前阶段: test
阶段状态: completed（或 blocked，如有🔴问题）
上次完成阶段: test
下一阶段: review
```

## 约束

- **禁止**只跑单元测试就声称测试完成
- **禁止**AC覆盖不全（必须100%覆盖）
- **禁止**测试结果只有“通过”而没有数据
- Strict模式**必须**执行性能测试

## 触发下一步

**输出测试报告后，必须等待用户确认**：

```markdown
测试完成，结果如下：
- 单元测试：XX/XX通过（覆盖率XX%）
- 集成测试：XX/XX通过
- AC覆盖：XX/XX通过

请确认或选择：
1. ✅ 测试通过，进入代码审查（6-review）
2. 🔧 有失败用例，回到开发修复（4-dev）
3. ✏️ 补充测试用例
```

- 用户确认测试通过 → 加载 `flow/stage-skills/stage-6-review/_SKILL.md`
- 测试不通过 / 用户要求修复 → 回到 stage-4-dev 修复问题

## 错误处理

- 测试失败 → 分析原因，分类为：代码bug/测试用例错误/环境问题
- 覆盖率不达标 → 补充缺失的测试用例
- AC无法验证 → 回到1-analysis澄清AC
- 性能不达标 → 优化代码或调整AC要求
