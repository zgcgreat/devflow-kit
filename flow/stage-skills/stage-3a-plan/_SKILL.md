## 执行流程

### Step 1: 判断项目类型

回答以下问题，决定后续章节：

1. 是否涉及数据库变更？ → 是/否
2. 是否新增API接口？ → 是/否
3. 是否使用中间件（Redis/MQ等）？ → 是/否
4. 是否有复杂算法？ → 是/否

### Step 2: 选择模板

**根据复杂度选择**：

- **精简版**（任务≤5个，中等复杂度）：
  - 读取 `@devflow-kit/flow/templates/03a-实施计划-精简版.md`
  - 约100行，核心设计+任务计划

- **完整版**（任务>5个，复杂项目）：
  - 读取 `@devflow-kit/flow/templates/03a-实施计划.md`
  - 约500行，含完整架构/部署/回滚

**询问用户**："选择模板类型：1️⃣ 精简版 2️⃣ 完整版"

### Step 3: 生成整体架构（仅完整版）

**1.1 技术方案概述**：
- 200-300字描述核心思路
- 技术选型（前端/后端/数据库/中间件）
- 关键架构决策及理由

**1.2 系统调用链路**（文本格式，禁止mermaid）：
```
客户端 → API Gateway → Service → Database
                    ↓
                  Redis
```

**1.3 API接口设计**（如问题2=是）：
- 接口清单表格
- 至少1个核心接口的完整文档（请求/响应/错误）

### Step 4: 生成数据层设计（仅完整版，如问题1=是）

**2.1 DDL**：
```sql
CREATE TABLE xxx (...);
CREATE INDEX idx_xxx ON xxx(...);
```

**2.2 ORM模型**：
```prisma
model Xxx { ... }
```

**2.3 缓存策略**（如使用Redis）：
- 缓存键格式
- TTL
- 更新策略
- 伪代码示例

### Step 5: 生成中间件配置（仅完整版，如问题3=是）

**表格形式**：
| 服务 | 用途 | 配置项 | 容错策略 |
|------|------|--------|---------|

**环境变量**：
| 变量名 | 类型 | 必填 | 默认值 | 说明 |

### Step 6: 为每个任务生成实施计划

**对03-tasks.md中的每个任务**：

#### 文件修改清单

| 文件 | 操作 | 说明 | 预估行数 |
|------|------|------|----------|

#### 核心实现

**函数签名**：
```typescript
interface XxxDTO { field: type; }
class XxxService { async method(): ReturnType; }
```

**伪代码**（如问题4=是，必须写）：
```typescript
async function xxx() {
  // 步骤1
  // 步骤2
}
```

**边界条件**：至少3个

#### 测试要点

- [ ] 正常场景
- [ ] 异常场景1
- [ ] 异常场景2
- [ ] 边界场景

#### 验收标准

- [ ] verify命令通过
- [ ] 功能符合预期
- [ ] 性能达标（如适用）

### Step 7: 生成风险与验证（仅完整版）

**主要风险**（至少3个）：
| 风险项 | 概率 | 影响 | 等级 | 缓解措施 |
|-------|------|------|------|---------|

**回归测试清单**：
- [ ] 核心AC全部通过
- [ ] 现有测试无退化
- [ ] 性能基准测试通过

### Step 8: 生成部署与回滚（仅完整版）

**部署步骤**：
1. 备份数据库
2. 执行迁移
3. 部署应用
4. 验证健康检查

**回滚方案**：
- 触发条件
- 回滚命令
- 预计时间

### Step 9: 自检清单

生成完成后逐项检查：

**精简版**：
- [ ] 技术方案概述清晰
- [ ] 根据Step 1的答案写了相应章节
- [ ] 每个任务有文件清单/核心实现/测试要点/验收标准
- [ ] 至少2个风险

**完整版**：
- [ ] 所有章节完整（架构/数据/中间件/任务/风险/部署）
- [ ] 调用链路是文本格式（非mermaid）
- [ ] 如有API，至少1个完整接口文档
- [ ] 如有数据库，有DDL+ORM模型
- [ ] 每个任务有完整的4个子章节
- [ ] 至少3个技术风险
- [ ] 部署和回滚步骤完整

### Step 10: 用户Review

输出：
```markdown
✅ 实施计划完成

请review以下关键点：
1. 整体架构设计是否符合预期？
2. 数据库/API设计是否正确？
3. 每个任务的实施方案是否清晰？
4. 风险评估是否全面？

确认后回复"确认"进入开发阶段，或指出需要调整的部分。
```

## 约束

- **禁止**照抄模板示例，必须根据实际项目填写
- **禁止**使用mermaid格式（渲染不出来）
- **必须**根据Step 1的判断决定是否写某章节
- 完整版**必须**包含所有7个章节

## 触发下一步

用户确认后：加载 `flow/stage-skills/stage-4-dev/_SKILL.md`

- 技术方案不明确 → 回到2-design澄清
- 用户要求调整 → 修改后重新review
