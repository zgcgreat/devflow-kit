> **前置条件**：无
> **后置产物**：`.specs/<req-id>/00-需求确认.md`

## Skill元信息

```yaml
name: stage-0-confirm

```

## 输入

- 用户原始需求描述
- `.specs/项目状态.md`（如存在）
- `.specs/上下文.md`（如存在）

## 输出

- `.specs/<req-id>/00-需求确认.md`
- 更新 `.specs/项目状态.md`

## 入口门禁

无（工作流起点）

## 执行流程

### Step 1: 读取项目状态

```markdown
IF .specs/项目状态.md 存在:
  读取当前阶段、已完成阶段
ELSE:
  创建新项目状态文件
```

**注意**：入场检测已由GO.md统一执行，本stage无需重复。

### Step 2: 需求澄清

使用 `idea-refine` skill 进行需求反问：
- 目标用户是谁？
- 核心功能是什么？
- 成功标准是什么？
- 有哪些约束条件？

### Step 3: 模式判定

根据需求复杂度判定执行模式：

| 维度 | Fast | Standard | Strict |
|------|------|----------|--------|
| 改动规模 | ≤2文件，<50行 | 中等功能 | 大型重构/高风险 |
| 风险等级 | 低 | 中 | 高 |
| 团队熟悉度 | 非常熟悉 | 一般 | 不熟悉/新技术 |

**输出格式**：
```markdown
🎯 模式推荐：<模式>（置信度 XX%）

评分详情:
- Fast: XX%
- Standard: XX%
- Strict: XX%

请确认或选择其他模式：
1. ✅ <推荐模式>
2. <备选1>
3. <备选2>
```

**注意**：模式推荐已由GO.md第五步统一输出，本stage只需记录用户选择的模式到产物中。

### Step 4: 生成产物

**⚠️ 强制规则**：输出前必须先读取模板文件 `flow/templates/00-需求确认.md`。

按模板生成 `.specs/<req-id>/00-需求确认.md`：
- **必须包含所有12个必填段落**（见模板L16-30）
- **所有 `<...>` 占位符必须替换为实际值**
- **不得省略或改写任何段落**

**自检**：输出前逐项核对模板强制规则（R13.9）

### Step 5: 更新项目状态

```markdown
当前阶段: confirm
阶段状态: completed
上次完成阶段: confirm
下一阶段: analysis
```

## 约束

- **禁止**跳过用户确认直接进入下一阶段
- **禁止**自行决定模式，必须让用户确认
- **必须**输出模式判定的评分详情和理由

## 自检清单

## 触发下一步

用户确认模式后：加载 `flow/stage-skills/stage-1-analysis/_SKILL.md`

## 错误处理

- 缺必需输入 → 反问用户补充
- 需求过于模糊 → 继续澄清直到可执行
- 用户要求跳过 → 记录跳过原因，允许但需警告风险
