## 执行流程

### Step 1: 扫描可用前置产物

**⚠️ 强制规则**：必须先扫描所有可能的前置产物，根据实际存在情况决定读取策略。

#### 1.1 扫描主流程产物

| 产物文件 | 存在性 | 优先级 | 提取内容 |
|---------|--------|--------|----------|
| STATE.md | ✅/❌ | 🔴 必须 | 活跃req、中断任务、当前阶段 |
| CONTEXT.md | ✅/❌ | 🟡 建议 | 技术栈、编码规范、禁动清单 |
| **memory/PROJECT_CONTEXT.md** | ✅/❌ | 🟡 建议 | **项目背景、历史决策、已知失败** |
| lessons-learned.md | ✅/❌ | 🟢 可选 | 历史教训、最佳实践 |

**扫描结果输出**：
```markdown
✅ 检测到 STATE.md → 提取：无活跃req
✅ 检测到 CONTEXT.md → 提取技术栈：Vue3 + TypeScript
✅ 检测到 memory/PROJECT_CONTEXT.md → 提取项目背景：B2C电商平台
✅ 检测到 memory/DECISIONS.md → 提取历史决策：2条
❌ 未检测到 lessons-learned.md → 使用通用最佳实践
```

#### 1.2 分级读取策略

**🔴 必须读取**（缺失会阻塞流程）：
- **STATE.md**：提供当前进度和中断任务
  - 如果缺失 → 创建新项目状态文件

**🟡 建议读取**（缺失采用降级策略）：
- **CONTEXT.md**：提供项目技术背景
  - 如果缺失 → 使用通用技术假设，在产物中增加“技术假设”章节
- **memory/PROJECT_CONTEXT.md**：提供项目历史和背景
  - 如果存在 → **自动加载**，AI会记住项目背景、历史决策、已知失败
  - 如果缺失 → 提示用户初始化记忆系统（`Use manage-memory`）

**🟢 可选读取**（补充信息）：
- **lessons-learned.md**：提供历史教训
  - 如果缺失 → 使用通用需求澄清问题

#### 1.3 降级策略

**如果某个产物不存在**：

1. **CONTEXT.md 缺失**：
   ```
   ⚠️ 警告：缺少CONTEXT.md，无法获取项目技术背景
   → 降级方案：在00-需求确认中增加"技术假设"章节
   → 询问用户："是否需要先运行 I-intel-scan 生成上下文？"
   ```

2. **lessons-learned.md 缺失**：
   ```
   ℹ️ 提示：缺少 lessons-learned.md，使用通用需求澄清问题
   → 提醒：建议在完成后将本次经验追加到 lessons-learned.md
   ```

#### 1.4 信息提取摘要

**输出格式**：
```markdown
### Step 1 输出：前置信息摘要

**已提取的关键信息**：

1. **项目层面**（来自项目状态）：
   - 活跃req：<无/REQ-XXX>
   - 中断任务：<无/TXX>

2. **技术层面**（来自上下文，如存在）：
   - 技术栈：<Vue3 + TypeScript>
   - 编码规范：<PascalCase组件名>

**下一步**：基于上述信息进行需求澄清
```

### Step 2: 需求澄清

**⚠️ 强制规则**：必须采用"单点聚焦 + 状态透明 + 进度可视化"的交互方式。

#### 2.1 列出所有澄清问题

基于用户需求和项目背景（CONTEXT.md），生成需求澄清问题清单（建议 3-5 个）。

**输出格式**：
```markdown
📍 当前阶段：0-confirm · 需求澄清
🎯 本步目标：确认 <需求核心点>

我需要确认以下 N 个问题，将逐个提问：

1. <问题1简述>
2. <问题2简述>
...

准备好了吗？我们从问题 1 开始 👇

---
**问题 1/N**：<第一个问题>
- A. <选项A>
- B. <选项B>
- C. <选项C>
- D. 其他（请说明）

请回复 A/B/C/D。
```

#### 2.2 逐个提问并等待回答

**执行流程**：
1. 输出完整问题清单 + 第一个问题（见 2.1）
2. **停等用户回答**
3. 记录用户答案
4. 输出下一个问题，格式：
   ```markdown
   ✅ 收到：<用户答案简述>

   ---
   **问题 2/N**：<第二个问题>
   ...
   
   请回复 A/B/C。
   ```
5. 重复直到所有问题回答完毕
6. 汇总所有答案，输出表格，进入 Step 3

**注意**：
- 每个问题后必须停等用户回复，不得一次性问完
- 如用户跳过某个问题，使用默认值并在产物中注明
- 问题数量建议 3-5 个，避免过多

### Step 3: 模式判定

**输出格式**（必须按此格式）：
```markdown
📍 当前阶段：0-confirm · 模式判定
🎯 本步目标：确定执行模式（Fast/Standard/Strict）

基于需求复杂度评估：

| 维度 | 评估 |
|------|------|
| 改动规模 | <评估结果> |
| 风险等级 | <评估结果> |
| 团队熟悉度 | <评估结果> |

🎯 建议模式：**<Standard / Strict>**  
理由：<简要说明为什么是这个模式>

📋 <模式> 完整流程：
<流程步骤简述>

请确认：
1. ✅ <建议模式>（推荐）
2. Fast
3. Strict

请回复 1/2/3。
```

**注意**：
- 必须包含"理由"字段，说明推荐该模式的原因
- 必须展示建议模式的完整流程和产物清单
- 其他模式只需一句话简介
- 用户可选择覆盖AI推荐

### Step 4: 完整性自检

**检查是否充分利用了前置产物**：

- ⏳ 是否读取了STATE.md？
- ⏳ 是否参考了CONTEXT.md中的技术背景（如存在）？
- ⏳ 对于缺失的产物，是否采用了合理的降级策略？
- ⏳ 需求澄清问题是否全面（目标用户/核心功能/成功标准/约束条件）？
- ⏳ 模式判定是否基于充分信息？

**如果发现遗漏**：
→ 回到Step 1重新读取
→ 或在本阶段产物中注明"因缺少XX产物，采用YY假设"

### Step 5: 读取模板并提取段落清单

**⚠️ 强制规则**：必须使用 `read_file` 工具读取模板文件。

```python

```

**从模板中提取必填段落清单**（模板 L16-30）：
```
□ Why（为什么做）
□ What（做什么）
□ 一句话总结
□ 背景与动机
□ 预期成果
□ 视觉调性（前端项目必填）
□ 影响面
□ 影响项表格
□ 范围排除（至少1条）
□ 验收线
□ 风险与未知
□ 架构层影响声明（如有）
```

**注意**：
- "架构层影响声明"如无内容，应保留段落标题并填写"无架构层影响"
- 不得因为"无内容"而删除该段落
