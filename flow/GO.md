# GO — devflow-kit 统一路由器（AI 内部文件）

> AI 指令（必须遵守）：
> 1. **本文件是唯一路由入口**。读完后按第三步路由表匹配用户意图，然后路由到对应 prompt
> 2. **第一条回复必须输出路由声明**（含选定的模式 Fast/Standard/Strict + 理由）
> 3. **所有产物必须保存到 `.specs/<req-id>/` 下**，不能放项目根目录
> 4. **不读本文件直接动手违反了流程规则**，会导致无入场扫描、无产物、无安全检查
> 5. **⚠️ 所有产物必须严格按模板输出**：输出前先读模板，产物包含模板所有段落，不得省略或改写（见 RULES.md R13.9）
>
> 用户使用方式：`Use devflow-kit` 或 `@flow/GO.md` + 一句话需求

> AI 看到本文件后按下方规则路由，自动决定阶段、自动生成 ID、自动按需加载工件，**不要等用户提供 ID 或路径**。

- 第五步 skill 未加载 → **禁止输出路由声明**
- 第七步未读模板 → **禁止输出产物**

## 模式与风险分级（走任何阶段前必读）

> **详细规则见 `mode-rules.md`**，本节仅保留核心要点。

进入流程前先判定 Fast / Standard / Strict 三档。用户已显式指定模式时可以沿用，但命中高风险矩阵必须升级。

| 模式 | 适用 | 是否停等用户 |
|------|------|--------------|
| **Fast** | 1~2 文件、<50 行、低风险、需求清楚 | 默认不必，但用户反对或发现隐藏风险时必须切换 |
| **Standard** | 常规 feature / bugfix、多文件、UI、业务逻辑 | 建议停等确认 |
| **Strict** | 高风险 / 生产敏感 / 架构或数据影响 | 必须停等确认 |

**高风险自动升级 Strict**：鉴权、权限、支付、schema/migration、公共 API、infra、部署、CI/CD、数据丢失风险、不可逆操作等。完整列表见 `mode-rules.md`。

**必须人工确认的门**：需求范围、技术栈方向、schema/migration、新依赖、删除文件、CI/CD 配置、生产发布。

## 红线·Token 预算（走任何阶段前必读）

> **详细规则见 `token-budget.md`**，本节仅保留核心要点。

**核心规则**：进入任何阶段的首轮消息，加载的 reference 总行数 ≤ **150 行**。

| 类型 | 路径 | 加载方式 |
|------|------|----------|
| SPEC（项目产物）| `.specs/<id>/*.md` | 整读 OK |
| REFERENCE（查阅型）| `flow/reference/*.md` | **禁止整读**，只读需要的节 |
| TEMPLATE / PROMPT | `flow/templates|prompts/*.md` | 整读 OK |

**何时需要预算估算**：Standard/Strict 模式、新需求、用户未显式指定模式时。详见 `token-budget.md`。

## 第一步 · 读取或创建项目状态（必须，跳过即违反）

1. 尝试读 `.specs/项目状态.md`。不存在 → 用 `flow/templates/项目状态.md` 模板创建空的 `.specs/项目状态.md`，然后继续

3. 如果存在 `中断任务` 非空 → **优先级最高**，直接走"恢复中断任务"分支（见下表）

### 可选 runtime adapter 检测

flow-kit 默认不依赖任何运行时。若项目同时存在 `.claude/hooks/forge-pretool-guard.ps1` 与 `.claude/hooks/forge-session-audit.ps1`，说明可选 Forge runtime adapter 已安装。

检测到 Forge 时，在路由声明里追加一行：

```text

```

## 第二步 · 老项目入场检测（brownfield 必跑）

> **详细规则见 `entry-check.md`**，本节仅保留核心要点。

### 强制门控

本步是进入任何需求相关阶段的前置门。未完成检测前，AI **禁止**进入 0-confirm / 1-analysis / 2-design / 文档解析模式。

**例外**：横向命令（A-architect / A-evolve / I-intel-scan / L-restyle / M-health）不跑本步。

### 检测执行步骤（必须按顺序执行）

```
步骤 2.1：检查 `.specs/项目状态.md` 是否有 ai_context_doc 字段
         ├─ 有 → 情况 A（直接用指定文档）
         └─ 无 → 继续

步骤 2.2：检查 `.specs/上下文.md` 是否存在
         ├─ 存在 + last_intel_scan < 90天 → 情况 B（直接用）
         ├─ 存在 + last_intel_scan > 90天 → 情况 C（提醒可重扫）
         └─ 不存在 → 继续

步骤 2.3：检查是否有其他 AI 上下文文档
         ├─ 有 → 情况 D（停等用户确认）
         └─ 无 → 继续

步骤 2.4：检查是否是 greenfield 项目
         ├─ 无 package.json / pyproject.toml / go.mod 等 → 情况 F（跳过）
         └─ 有 → 情况 E（停等用户确认）
```

### 强制检查点：输出检测结果框

**在输出路由声明前，必须先输出检测结果框**：

```
┌─────────────────────────────────────────────────────────────┐
│  🔍 入场检测（第二步 · 必须）                                  │
├─────────────────────────────────────────────────────────────┤
│  检测结果：<情况 A/B/C/D/E/F>                                 │
│  项目类型：<brownfield / greenfield>                         │
│  上下文文档：<路径 或 "无">                                   │
│  入场扫描状态：<已完成 / 已跳过 / 待用户确认>                  │
└─────────────────────────────────────────────────────────────┘
```

### 各情况处理规则

| 情况 | 判定条件 | AI 动作 | 是否停等用户 |
|------|----------|---------|--------------|
| **A** | 项目状态.md 有 ai_context_doc 字段 | 直接读指定文档 | ❌ 不停等 |
| **B** | 上下文.md 存在 + 扫描 < 90天 | 直接读上下文.md | ❌ 不停等 |
| **C** | 上下文.md 存在 + 扫描 > 90天 | 读上下文.md + 提醒可重扫 | ❌ 不停等 |
| **D** | 发现其他 AI 上下文文档（AGENTS.md 等） | 输出选项让用户选 | ✅ **必须停等** |
| **E** | brownfield + 无任何上下文文档 | 输出选项让用户选 | ✅ **必须停等** |
| **F** | greenfield（无项目文件） | 跳过检测 | ❌ 不停等 |

**⚠️ 禁止行为**：
- 情况 D/E 时 AI 自行决定跳过 → **违规**
- 未输出检测结果框就进入后续阶段 → **违规**
- 检测结果为"待用户确认"时继续执行 → **违规**

## 第三步 · 路由表（匹配用户意图 → 阶段）

### 路由匹配执行步骤

```
步骤 3.1：检查是否有「中断任务」
         ├─ 项目状态.md 有中断任务 + 用户说"继续" → 恢复中断任务
         └─ 无中断任务 → 继续

步骤 3.2：检查用户是否要求跳过流程
         ├─ 用户说"不要走流程"/"就直接帮我改" → 直接做，跳过 flow-kit
         └─ 无跳过要求 → 继续

步骤 3.3：检查用户意图类型
         ├─ "健康检查"/"项目状态" → M-health
         ├─ "evolve"/"architect"/"梳理架构" → A-evolve 或 A-architect
         ├─ "这段代码怎么回事"/纯技术问题 → 直接回答，不用 flow-kit
         ├─ 上传文档/PRD/设计稿/技术方案 → 文档解析模式
         └─ 有新想法想做件事 → 继续

步骤 3.4：检查需求状态
         ├─ .specs/ 下无活跃 req → 0-confirm
         ├─ .specs/ 下有活跃 req → 按中断处理规则
         └─ 有非活跃 req → 先归档旧 req，再走 0-confirm
```

### D. 文档解析模式（用户上传已有文档时触发）

> **详细规则见 `doc-parse.md`**，本节仅保留核心要点。

当用户上传或粘贴需求文档、设计稿、技术方案时，跳过反问式澄清，直接解析文档内容。

**⚠️ 文档解析模式仍需遵守以下必选步骤：**

1. **第一步（项目状态）**：必须执行，读取或创建 `.specs/项目状态.md`
2. **第二步（入场检测）**：brownfield 项目必须执行，检测 `上下文.md` / AI 上下文文档
3. **模式确认**：解析完成后，必须判定 Fast/Standard/Strict 模式并等用户确认

**触发类型**：

| 触发语 | 解析类型 |
|--------|----------|
| "这是需求文档"、"PRD 见附件"、粘贴大段需求描述 | 需求文档解析 |
| "这是设计稿"、上传图片、Figma 链接 | 设计稿解析 |
| "这是技术设计"、"架构方案见附件" | 技术方案解析 |

**强制执行顺序**：
- Step 2 未输出检测结果框 → 禁止执行解析
- 检测结果为"待用户确认"（情况 D/E）→ 禁止继续，必须等用户回复

**产物输出**：
- 所有文档解析模式**必须先生成 `00-需求确认.md`**，不得跳过
- **禁止 AI 自行编造产物格式**：必须严格按模板输出

| 用户说 | 路由到 | 说明 |
|---|---|---|
| "换个方案" / "换设计" | 回到 2-design | 触发新 ADR，生成设计文档变体 |

| "继续" / "下一阶段" / "继续下一阶段" | 按 R1.5 执行下一阶段 | 不阻塞，直接往下走 |

### C. 阶段门验证（防跳级 · 路由到非 0-confirm 阶段时必查）

> **详细规则见 `gate-rules.md`**，本节仅保留核心要点。

**触发条件**：路由目标 ≠ 0-confirm（即用户意图已跳过了前面的阶段）。

**核心规则**：缺前置产物 → 不允许直接进入，必须先补跑缺失阶段。

**阶段依赖**：`0-confirm → 1-analysis → 2-design → [2a-ui-design] → 3-task → 4-dev → 5-test → 6-review → 7-integration`

**Fast 模式例外**：允许跳过 0~3 阶段直接进入 4-dev，但必须：
1. AI 已明确判定为 Fast 模式
2. 改动 ≤ 2 文件、< 50 行、低风险
3. 用户未反对
4. 路由声明中必须包含 verify 命令

**用户显式跳过**：允许，但 AI 必须在路由声明中记录跳过内容和原因。

## 自检（产出路由声明前）

> ⚠️ **顺序要求**：自检必须按以下顺序执行。前一项不通过，后一项禁止执行。

### 第一门控（强制）— 入场检测

| 检查项 | 通过标准 | 不通过处理 |
|--------|----------|------------|
| 入场检测已输出 | 已输出检测结果框（含情况判定、项目类型、上下文文档、入场扫描状态） | **禁止继续**，回到第二步 |
| 入场检测结果 | 为"已完成"或"已跳过"或"greenfield" | 情况 D/E → **禁止继续**，等用户确认 |

**自检命令**：
```
□ 我已输出检测结果框？ → 是/否
□ 检测结果是 A/B/C/F？ → 是/否（D/E 需等用户）
```

### 第二门控（入场检测通过后）— 项目状态

| 检查项 | 通过标准 | 不通过处理 |
|--------|----------|------------|
| 项目状态.md | 已读取（如存在） | 读取后再继续 |
| 意图匹配 | 已按路由表匹配 | 重新匹配 |

**自检命令**：
```
□ 我已读 .specs/项目状态.md？ → 是/否/不存在
□ 我已按路由表确定目标阶段？ → 是/否
```

### 第三门控（路由目标 ≠ 0-confirm 时）— 阶段门

| 检查项 | 通过标准 | 不通过处理 |
|--------|----------|------------|
| 前置产物检查 | 所有前置产物存在 | 补跑缺失阶段 |
| 用户跳过确认 | 用户显式确认跳过 | 停等用户确认 |

**自检命令**：
```
□ 路由目标是 0-confirm？ → 是（跳过本门控）/ 否（继续检查）
□ 前置产物都存在？ → 是/否（列出缺失）
□ 缺失阶段已补跑或用户已确认跳过？ → 是/否
```

### 第四门控（模式确认）

| 检查项 | 通过标准 | 不通过处理 |
|--------|----------|------------|
| 模式判定 | 已判定 Fast / Standard / Strict | 按规则判定 |
| Fast 确认 | 可直接声明 | — |
| Standard 确认 | 建议停等用户确认 | 输出模式确认，等用户 |
| Strict 确认 | 必须停等用户确认 | 输出模式确认，等用户 |

**自检命令**：
```
□ 我已判定模式？ → Fast / Standard / Strict
□ 判定理由：<理由>
□ Standard/Strict 已等用户确认？ → 是/否（Fast 跳过）
```

### 第五门控（加载检查 · 关键）

| 检查项 | 通过标准 | 不通过处理 |
|--------|----------|------------|
| skill 加载 | 所有指定 skill 已全读 `_SKILL.md` | **禁止输出路由声明** |
| reference 加载 | 按节读取，已列出节名和行数 | 补加载后再声明 |
| Token 预算 | reference 总行数 ≤ 150 | 拆分到后续加载 |
| 未越界 | 未整读「查表」或「按需」列文件 | 回滚越界读取 |

**自检命令**：
```
□ 我已加载所有必需 skill？ → 是（列出）/ 否
□ 我已按节读取 reference？ → 是（列出节名+行数）/ 否
□ reference 总行数 ≤ 150？ → 是/否
□ 我没有越界整读？ → 是/否
```

### 第六门控（路由声明完整性）

| 检查项 | 通过标准 | 不通过处理 |
|--------|----------|------------|
| 七要素齐全 | 模式/预估/skill/reference/工件/未加载/第一动作 | 补充缺失要素 |
| reference 声明 | 已列出节名和行数 | 补充声明 |
| ID 生成 | 已自动生成 Req-ID | 生成 ID |

**自检命令**：
```
□ 路由声明包含七要素？ → 是/否（列出缺失）
□ 我没有要求用户提供 ID/路径？ → 是/否
```

## 产物输出前自检（产物输出前必须执行）

> ⚠️ **此检查在路由声明后、产物输出前执行。**

### 第七门控（模板校验 · 产物输出前）

| 检查项 | 通过标准 | 不通过处理 |
|--------|----------|------------|
| 模板已读 | 已读取对应模板文件 | **禁止输出产物** |
| 必填段落清单 | 逐项对照模板开头的清单，全部已包含 | 补齐缺失段落 |
| 结构完整 | 产物包含模板所有必填段落 | 补齐缺失段落 |
| 字段填充 | 占位符已替换为实际值 | 替换占位符 |

**强制执行步骤**：
```
步骤 7.1：读取模板文件（如 flow/templates/上下文.md）
步骤 7.2：找到模板开头的「必填段落清单」
步骤 7.3：逐项检查产物是否包含每个段落
步骤 7.4：任一项缺失 → 补齐后再输出
步骤 7.5：所有段落齐全 → 输出产物
```

**⚠️ 特别注意：上下文.md 和 项目状态.md**
这两个文件经常被 AI 跳过模板直接输出。必须：
1. 先读取 `flow/templates/上下文.md` 或 `flow/templates/项目状态.md`
2. 对照必填段落清单逐项确认
3. 确保包含所有必填段落后再输出

**自检命令**：
```
□ 我已读取目标模板？ → 是（模板路径）/ 否
□ 我已找到模板的「必填段落清单」？ → 是/否
□ 我已逐项对照清单？ → 是（列出所有段落状态）/ 否
□ 产物包含所有模板段落？ → 是/否（列出缺失）
□ 占位符已替换？ → 是/否（列出未替换的）
```

**⚠️ 禁止行为**：
- 未读模板就输出产物 → **违规**
- 未对照必填段落清单 → **违规**
- 产物遗漏模板段落 → **违规**
- 保留尖括号占位符 → **违规**

## 违规处理协议

检测到即将违反任一规则时：

1. **立即停止**当前动作
2. **输出违规声明**：
   ```
   ⚠️ 规则 R{编号} 触发：<原因>
   
   需要人工决策。请确认如何处理：
   1. <选项 A>
   2. <选项 B>
   ```
3. **等待用户回复**，禁止"自我授权"绕开规则
