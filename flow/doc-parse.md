# 文档解析模式详细规则

> 本文件是 GO.md 文档解析部分的详细展开，由 GO.md 按需加载。

## 触发条件

当用户上传或粘贴需求文档、设计稿、技术方案时，跳过反问式澄清，直接解析文档内容。

| 触发语 | 解析类型 |
|--------|----------|
| "这是需求文档"、"PRD 见附件"、粘贴大段需求描述、上传 .md/.doc/.pdf | 需求文档解析 |
| "这是设计稿"、上传图片、提供 Figma/Sketch 链接 | 设计稿解析 |
| "这是技术设计"、"架构方案见附件" | 技术方案解析 |

---

## 强制前置步骤

**⚠️ 文档解析模式仍需遵守以下必选步骤：**

1. **第一步（项目状态）**：必须执行，读取或创建 `.specs/项目状态.md`
2. **第二步（入场检测）**：brownfield 项目必须执行，检测 `上下文.md` / AI 上下文文档
3. **模式确认**：解析完成后，必须判定 Fast/Standard/Strict 模式并等用户确认

---

## 通用流程

```
┌─────────────────────────────────────────────────────────────┐
│  📋 文档解析通用流程                                          │
├─────────────────────────────────────────────────────────────┤
│  Step 1: 读取项目状态（第一步 · 必须）                        │
│     └─ 读/创建 `.specs/项目状态.md`                          │
│                                                              │
│  Step 2: 入场检测（第二步 · 强制门控）                        │
│     ├─ ⚠️ 本步是强制门控，未完成禁止继续                      │
│     ├─ 检测 `上下文.md` / AI 上下文文档                        │
│     ├─ 输出检测结果框（含情况判定、项目类型、上下文文档、状态）│
│     └─ 新项目（greenfield）可跳过，但必须输出检测结果框       │
│                                                              │
│  Step 3: 解析文档内容（入场检测通过后）                       │
│     └─ 提取结构化信息                                        │
│     └─ 评估文档质量                                          │
│     └─ 输出解析结果                                          │
│                                                              │
│  Step 4: 模式判定与确认（必须停等用户）                       │
│     └─ 根据文档内容判定 Fast/Standard/Strict                 │
│     └─ 输出模式确认，等用户回复                               │
│                                                              │
│  Step 5: 用户确认后继续                                      │
│     └─ 生成产物 / 进入下一阶段                                │
└─────────────────────────────────────────────────────────────┘
```

**⚠️ 强制执行顺序**：
- Step 2 未输出检测结果框 → 禁止执行 Step 3
- 检测结果为"待用户确认"（情况 D/E）→ 禁止继续，必须等用户回复

## 文档解析失败的处理

如果文档无法解析（格式不支持、内容混乱、语言不通等），输出：

```
┌─────────────────────────────────────────────────────────────┐
│  ❌ 文档解析失败                                              │
├─────────────────────────────────────────────────────────────┤
│  原因：<具体原因，如"格式不支持"/"内容混乱无法提取结构"等>    │
│                                                              │
│  请选择：                                                     │
│  [1] 重新上传文档（支持 .md/.txt/.json 格式）                │
│  [2] 改用文字描述需求（进入 0-confirm 反问模式）              │
│  [3] 手动提供关键信息（我会引导你填写）                       │
└─────────────────────────────────────────────────────────────┘
```

## D.1 需求文档/PRD 解析

**执行流程**：

```
Step 1-2: 通用流程（见上）

Step 3: 解析文档 → 提取结构化内容
   - 背景/动机
   - 功能列表
   - 验收标准（如有）
   - 非功能性需求（如有）
   - 范围边界（如有）

Step 4: 评估文档完整度 → 输出评估结果

┌─────────────────────────────────────────────────────────────┐
│  📄 需求文档解析结果                                          │
├─────────────────────────────────────────────────────────────┤
│  ✅ 已提取（清晰部分）：                                       │
│     - 背景：<从文档提取>                                      │
│     - 功能：<列表>                                           │
│     - AC：<如有，逐条列出>                                    │
│                                                              │
│  ⚠️ 需澄清（歧义/模糊）：                                     │
│     - [Q1] <原文> — <具体问题>                               │
│     - [Q2] <原文> — <具体问题>                               │
│                                                              │
│  ❌ 缺失（应该有但没有）：                                     │
│     - [M1] <缺失项>                                          │
├─────────────────────────────────────────────────────────────┤
│  📊 文档质量：<高/中/低>（评估标准见 D.4）                     │
│  🎯 建议路径：                                                │
│     • 高质量（AC 齐全无歧义）→ 直接进入 1-analysis            │
│     • 中质量（有歧义需澄清）→ 回答问题后进入 1-analysis        │
│     • 低质量（多处缺失）→ 建议补充文档或走 0-confirm 反问模式  │
├─────────────────────────────────────────────────────────────┤
│  🎯 建议模式：<Fast/Standard/Strict>                         │
│     理由：<根据文档内容判定>                                  │
│     流程：<该模式对应的最小流程>                              │
│                                                              │
│  请选择：                                                     │
│  [1] 回答上述问题后继续（推荐）                               │
│  [2] 我补充/修改文档后重新上传                                │
│  [3] 按当前内容继续，缺失项用默认值（会标注风险）              │
│  [4] 改走 0-confirm 反问模式（重新梳理需求）                  │
│  [5] 我对解析结果不满意，手动修正                              │
└─────────────────────────────────────────────────────────────┘

Step 5: 用户确认模式后，根据选择继续：
   - 选 1 → 用户回答问题 → 生成产物 → 进入下一阶段
   - 选 2 → 等待新文档 → 重新解析
   - 选 3 → 用默认值填充缺失项 → 标注风险 → 生成产物
   - 选 4 → 进入 0-confirm 正常流程
   - 选 5 → 进入手动修正模式（见 D.5）
```

**自动生成 req-id**：从文档标题或核心功能提取，规则同 0-confirm 步骤 1。

**产物输出**：
- **⚠️ 强制要求**：所有文档解析模式**必须先生成 `00-需求确认.md`**，不得跳过
- 高质量文档：生成 `00-需求确认.md` 后，可直接进入 `1-analysis` 生成 `01-需求分析.md`
- 中/低质量文档：生成 `00-需求确认.md`，澄清问题后再生成 `01-需求分析.md`
- **禁止 AI 自行编造产物格式**：必须严格按 `@devflow-kit/flow/templates/00-需求确认.md` 模板输出

**执行流程**：

```

   - 页面结构
   - 组件清单
   - 交互说明（如有）
   - 样式/颜色/字体（如有）

Step 4: 输出解析结果 + 澄清问题

┌─────────────────────────────────────────────────────────────┐
│  🎨 设计稿解析结果                                            │
├─────────────────────────────────────────────────────────────┤
│  ✅ 已识别：                                                  │
│     - 页面：<页面名称>                                        │
│     - 组件：<组件列表>                                        │
│                                                              │
│  ⚠️ 需确认：                                                  │
│     - [Q1] <组件> 的交互行为？                                │
│     - [Q2] 响应式断点？                                       │
├─────────────────────────────────────────────────────────────┤
│  🎯 建议模式：<Fast/Standard/Strict>                         │
│     理由：<根据设计稿复杂度判定>                              │
│                                                              │
│  建议路径：                                                   │
│  [1] 已有需求文档 → 回答问题后进入 2a-ui-design               │
│  [2] 需要先梳理需求 → 进入 0-confirm 确认需求范围             │
│  [3] 我对解析结果不满意，手动修正                              │
└─────────────────────────────────────────────────────────────┘

Step 5: 用户确认模式后继续
```

**产物输出**：
- 设计稿解析本身不直接生成产物
- **⚠️ 强制要求**：必须先走 `0-confirm` 生成 `00-需求确认.md`，再进入 `1-analysis`
- 需要先有 `01-需求分析.md`，才能进入 `2a-ui-design`
- **禁止 AI 自行编造产物格式**：必须严格按模板输出

**完整路径**：
```
设计稿解析 → 进入 0-confirm 生成 00-需求确认.md → 1-analysis 生成 01-需求分析.md → 进入 2a-ui-design
```

```
│  [4] 我对解析结果不满意，手动修正                              │

```

**产物输出**：
- **⚠️ 强制要求**：所有路径**必须先生成 `00-需求确认.md`**，不得跳过
- 路径 [1]：生成 `00-需求确认.md` 后，直接生成 `02-方案设计.md`（方案对齐）
- 路径 [2]：生成 `00-需求确认.md` → 生成 `01-需求分析.md` → 生成 `02-方案设计.md`
- 路径 [3]：走完整流程（00 → 01 → 02）
- **禁止 AI 自行编造产物格式**：必须严格按模板输出

**注意**：技术方案解析时，AI 必须在 `00-需求确认.md` 顶部标注：
```markdown
> ⚠️ 本需求确认书由用户上传的技术方案文档解析生成。
> 如发现需求不清晰，应暂停并回到 0-confirm 重新确认。
```

## D.4 文档质量评估标准（量化指标）

| 等级 | 量化标准 | 处理方式 |
|------|----------|----------|
| **高** | AC 数量 ≥ 3 且每条有 Given/When/Then；范围边界明确（v1/v2/out 都有）；非功能性需求 ≥ 2 项 | 直接生成 01-需求分析.md，进入下一阶段 |
| **中** | 有功能描述但 AC < 3 或格式不完整；范围边界部分缺失；非功能性需求 < 2 项 | 针对性反问 1-2 轮，补全后生成产物 |
| **低** | 只有笼统描述，无 AC；范围边界缺失；无非功能性需求；多处歧义 | 建议补充文档，或走 0-confirm 反问模式 |

**评估检查清单**：

```
□ AC 数量：≥ 3 / 1-2 / 0
□ AC 格式：完整 Given/When/Then / 部分完整 / 无结构
□ 范围边界：v1/v2/out 都有 / 部分缺失 / 完全缺失
□ 非功能性需求：≥ 2 项 / 1 项 / 无
□ 歧义点：0 / 1-3 / > 3
```

**关键原则**：
- **先提取再反问**：不是从头问，只问文档不清楚的点
- **区分"需澄清"和"缺失"**：澄清 = 文档有但模糊；缺失 = 文档没有但应该有
- **给用户选择权**：可以回答问题、补充文档、接受默认、或改走反问模式
- **不强制重写文档**：低质量时用户可选择"回答问题"而非重写

## D.5 手动修正模式

当用户选择"我对解析结果不满意，手动修正"时：

```
┌─────────────────────────────────────────────────────────────┐
│  ✏️ 手动修正模式                                              │
├─────────────────────────────────────────────────────────────┤
│  请告诉我需要修正的内容：                                     │
│                                                              │
│  1. 修正已提取的内容：                                        │
│     - 背景/动机：<用户输入>                                   │
│     - 功能列表：<用户输入>                                    │
│     - AC：<用户输入>                                          │
│                                                              │
│  2. 补充缺失的内容：                                          │
│     - <用户输入>                                              │
│                                                              │
│  3. 删除错误提取的内容：                                      │
│     - <用户输入>                                              │
│                                                              │
│  输入完成后回复"确认"，我将重新生成解析结果。                 │
└─────────────────────────────────────────────────────────────┘
```

用户确认后，AI 重新输出解析结果，并再次询问模式确认。
