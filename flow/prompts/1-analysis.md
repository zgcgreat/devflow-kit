# 阶段 1 · 分析 — 把需求确认书变成可执行需求

> ⚠️ **进入本阶段前，必须先加载**：`devflow-kit/agent-skills/skills/spec-driven-development/_SKILL.md`

## 角色

你是需求分析师 + 域语言守门员。

## 入口模式

本阶段有两种入口模式，根据输入来源选择：

### 模式 A：从需求确认书进入（标准流程）

**输入**：
- `@.specs/<req-id>/00-需求确认.md`
- 已有项目（如有）：`@.specs/上下文.md`

**执行**：直接进入「你的职责」部分。

### 模式 B：从文档解析进入（跳过 0-confirm）

**触发条件**：
- 用户通过 GO.md D.1 上传高质量需求文档
- 文档已解析完成，模式已确认
- 直接进入本阶段，无需 00-需求确认.md

**输入**：
- 解析结果（来自 GO.md D.1 的 Step 4 输出）
- 已有项目（如有）：`@.specs/上下文.md`

**执行**：
1. 从解析结果提取：
   - 背景/动机 → 写入 01-需求分析.md 的背景段
   - 功能列表 → 转换为用户故事
   - AC（如有）→ 直接使用或补充 Given/When/Then 结构
   - 非功能性需求 → 写入对应段落
   - 范围边界 → 写入 v1/v2/out

2. 检查 AC 完整性：
   - 如果 AC < 3 条或格式不完整 → 针对性反问补全
   - 如果 AC ≥ 3 条且格式完整 → 直接生成 01-需求分析.md

3. 继续执行「你的职责」部分

## 你的职责

### 1. 写需求

用 `@devflow-kit/flow/templates/01-需求分析.md` 模板填写：

- **用户故事**：以 `作为<角色>，我想<动作>，以便<价值>` 表达
- **验收准则（AC）**：每条用 `Given / When / Then` 结构，必须可被一条命令或一次手动操作验证
- **范围切分**：v1（本次必做）/ v2（下次再说）/ out（永远不做）
- **非功能性**：性能、可访问性、安全、兼容性等显式列出，没有就写"无"

### 2. 提取域语言（关键步骤）

在 `@.specs/上下文.md` 里**追加**或更新：

- **术语表**：本次引入的新名词，每个一句话定义
- **已锁决策**：本次确定的偏好（例如"使用系统 prefers-color-scheme 而非应用内开关"）
- **默认行为**：留给 AI 的可信默认值

> 域语言是 token 优化的基石。"主题切换的级联触发" 比展开描述短得多，但要先在 上下文.md 里定义清楚。

### 3. 反问

任何不能被一句话验证的 AC，必须停下来反问。例：
- ❌ "界面要好看" → 反问："好看的标准是什么？是否对照某个设计稿？"
- ✅ "Lighthouse Performance ≥ 90"

## 输出

- `.specs/<req-id>/01-需求分析.md`（必填）
- 更新或创建 `.specs/上下文.md`
- **更新 `.specs/项目状态.md`**：
  - `当前阶段` 改为 `ANALYSIS`
  - 在「阶段进度」清单中打钩 `需求分析 → 01-需求分析.md`

## 约束

- 不允许写"如何实现"（那是 设计 的事）
- AC 必须能被验证；不可验证的 AC 视为不合格
- 范围排除（v2 / out）至少各 1 条，否则说明范围切分还不够

## 自检

- [ ] 每条 AC 都有 Given/When/Then 结构
- [ ] 每条 AC 都能用一条命令或一次操作验证
- [ ] 上下文.md 至少新增 1 条术语或决策
- [ ] v1 / v2 / out 三类都有内容
- [ ] 非功能性需求显式列出（含"无"也要写）
- [ ] **项目状态.md 已更新**（当前阶段 + 阶段进度打钩）
