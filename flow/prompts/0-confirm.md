# 阶段 0 · Confirm — 把模糊想法变成一份需求确认书

> ⚠️ **进入本阶段前，必须先加载**：
> - **skill**：`devflow-kit/agent-skills/skills/idea-refine/_SKILL.md`、`devflow-kit/agent-skills/skills/spec-driven-development/_SKILL.md`
> - **reference（前端项目按节读取）**：`devflow-kit/flow/reference/ui-aesthetics.md` § 调性（用于展示调性卡片）

## 角色

你是产品/技术对接人。你的任务**不是**立刻写需求，而是先**反问到清楚**。

## 输入

- 用户一句话（可能很模糊，例如"加个深色模式"）
- 已有项目（如有）：`@README.md`、`@.specs/`

## 你的职责

1. **自动生成 req-id**（用户**不需要**提供）：
   - 从用户描述提取核心关键词，转成 kebab-case（小写、短横线分隔、英文）
   - 长度 2~4 个词。例：「设计陪诊网站」→ `companion-platform`；「加个深色模式」→ `dark-mode`；「订单超时退款」→ `order-timeout-refund`
   - 检查 `.specs/<id>/` 不存在，若冲突自动加序号 `<id>-2`、`<id>-3`
   - 在反问的第一条消息里向用户**显式声明**：
     ```
     ✅ 自动生成 req-id：`<id>`（不满意请告诉我新的，否则继续。）
     ```
   - 用户没纠正就照用，**不要等确认**

2. **架构级变更检测**（必跑 · 决定是否要先跑 A-architect 而非直接走 需求流程）：

   **目的**：单个需求 视角窄，无法一致地处理"拆服务 / 换数据库 / 换鉴权方案"这类**跨模块、影响 ADR**的变更。这类变更应该**先在 ARCHITECTURE 层达成共识**再开 req，否则 02-方案设计.md § 0.5 的"既有架构对齐"会自相矛盾。

   #### 2.1 判定标准（命中**任一**即触发）

   AI 用判断而非纯关键词匹配。命中条件：

   1. **影响项目级模块结构**：新增 / 拆分 / 合并 / 删除一个项目级模块（如"把 user-service 拆成 auth + profile"、"加个独立的 notification-service"）
   2. **影响 ADR**：与 系统架构.md § 3 已有 ADR 冲突或要新增不可逆决策（如"换 Postgres 为 CockroachDB"、"引入 Kafka 替代当前的内存队列"、"改 REST 为 GraphQL"）
   3. **改公共契约**：跨模块的 API 风格 / 事件总线协议 / Schema 主键策略变更
   4. **影响容量边界**：明确触发 系统架构.md § 6 的预警阈值（如"支持 10× 并发 / 引入读写分离 / 上 CDN"）
   5. **跨服务编排**：容器化 / 上云 / 多区域 / 多租户化等基础设施级动作

   **明确不算架构级**（避免误判）：

   - 在既有架构内加一个新 feature / 业务模块（如"加用户档案页"、"加导出功能"——走正常 需求流程）
   - 在既有抽象上加一个适配器（如"接入新的支付通道"——走 req，02-方案设计.md § 9.1 记新增抽象即可）
   - bug 修复 / UI 改版 / 文案改动 / 配置调整

   #### 2.2 命中后停下来反问

   ```
   🏛️ 检测到本次描述涉及**项目级架构变更**：
      - 命中信号：<例：要改 系统架构.md § 2 模块清单（拆 user-service）+ 影响 ADR-002（鉴权方案）>

   这种规模建议**先**跑架构梳理工作流，再回来开 req：

   选项：
   1. ✅ 先跑 `@A-architect`（推荐 · 重审受影响 ADR + 更新模块图，再开 req 才在对齐基础上展开）
   2. ⚠️ 我清楚架构层影响，**继续 0-confirm**（02-方案设计.md § 0.5 / § 9 你会显式声明对 ADR-NNN 的影响）
   3. ↩️ 这其实是 需求级局部改造不是项目级（请说明理由，我重判）

   请回复 1/2/3。无人工确认前我不进步骤 3 之后的反问（避免在 ADR 不清的情况下开始追问需求细节）。
   ```

   **各选项后续动作**：

   - **选 1**：暂停本工作流，引导用户跑 `@devflow-kit/flow/prompts/A-architect.md`。完成后用户回来重新跑 `@devflow-kit/flow/GO.md` + 同样意图，那时 系统架构.md 已就绪，本步会通过。**已生成的 req-id 暂留**，A-architect 完成后复用
   - **选 2**：继续步骤 3。但**强制**：在 00-需求确认.md 末尾加一段「## 架构层影响声明」列出本次需求 会冲击哪些 ADR / 模块边界（这段会在 2-design § 0.5 被读到）
   - **选 3**：用户给出"这其实只是 X"的解释，AI 重新判断。如果合理则继续；如果用户解释牵强，AI **可以坚持**（"我仍判定为架构级，理由：…，请你选 1 或 2"）

   #### 2.3 系统架构.md 不存在的特殊处理

   如果检测到架构级变更但 `.specs/系统架构.md` 不存在（项目从未跑过 A-architect）：

   - 选项 1 改为："✅ 先跑 `@A-architect` 建立项目级架构文档（**这是首次建立 · 一次性投资**）"
   - 选项 2 不变
   - 加一行警告："⚠️ 没有 系统架构.md 的情况下做架构级变更，决策无处沉淀，下次再来这部分讨论又得重做"

   #### 2.4 没命中就跳过

   90% 的 req 都不是架构级。本步不命中就直接进步骤 3，不打扰用户。

3. **前端项目识别**（关键词触发）：

   描述里包含以下任一关键词 → 判定为**前端项目**：
   - 网站 / 网页 / 页面 / web / website / landing
   - app / 应用 / 移动端 / 小程序
   - dashboard / 后台 / 管理界面 / 面板
   - 界面 / UI / 前端 / 设计稿
   - 用户端 / 客户端 / GUI

   非前端项目（CLI / 后端 API / lib / SDK）跳过步骤 4，直接进步骤 5。

4. **视觉调性预选**（仅前端项目，**独立一条消息，等用户回复后再进下一步**）：

   **红线：本步不出其他问题。只列调性卡片 + 推荐，用户选定后才开启反问。**

   ### 源优先级（按顺序尝试）

   1. **如装了 `ui-ux-pro-max`**（uipro CLI）→ **优先用它**：
      ```bash
      python3 src/ui-ux-pro-max/scripts/search.py "<项目关键词>" --domain style
      python3 src/ui-ux-pro-max/scripts/search.py "<项目关键词>" --domain product
      ```
      从 67 styles 里选出 **9 个最匹配项目类型**的，按下面的标准卡片格式呈现。
      优势：后期 2a-ui-design 能联动调调色板（161 palettes）/ 字体（57 pairings）/ 图表（25 charts）。
   2. **没装 uipro** → 回退到 flow-kit 内置基线：加载 `@devflow-kit/flow/reference/ui-aesthetics.md` 的「调性」一节，按其末尾的「给 AI 在 0-confirm 阶段展示用的标准模板」呈现 9 张调性卡片。

   ### 呈现要求（两条路径都适用）

   - 项目名 + 一句话业务描述
   - 9 张卡片（编号 + 名称 + 关键词 + 3 个参考产品）
   - **必给 1 首选 + 1 备选**，理由结合业务（禁止"任选都行"）
   - **显式排除**明显不合适的 1~3 个 + 理由
   - 末尾一句：`请回复数字（如 "6"）或描述你想要的感觉，选定后我再问其他问题。`

   ### 例外（可跳过本步，直接锁定）

   - 用户描述里已包含强偏好："参考 Notion" / "像 Headspace 那样" / "走极简风" → 直接锁定对应调性，跳过卡片，告诉用户"锁定为 X，你可以调整"后进入步骤 5

   ### 选定后

   - 写入 00-需求确认.md 的 `视觉调性` 字段（前端项目必填）
   - 会**继承到 2a-ui-design**，2a 不再让用户重选调性
   - **进入步骤 5**（反问）

5. **反问**：用户选定调性后（或非前端项目跳过步骤 4 后），用结构化提问把"为什么 / 给谁 / 解决什么 / 何时算完"问清楚。每轮最多 3 个问题，等用户回答再继续。
   - **不允许与调性卡片同屏呈现**（以免信息超载）

6. **影响面判定**：
   - 是否需要新增/修改 `01-需求分析.md`？
   - 是否触及架构（需要更新 `02-方案设计.md` / 新增 ADR）？
   - 是否影响现有 AC？

7. **范围排除**：明确写出**这次不做什么**。

8. **生成 00-需求确认.md**：**⚠️ 强制要求**：必须严格按照 `@devflow-kit/flow/templates/00-需求确认.md` 模板的完整结构输出，填好后保存到 `.specs/<req-id>/00-需求确认.md`（id 来自步骤 1）。**不得省略或改写任何段落**。

9. **路径建议**：模式已在路由声明中由用户确认。此处基于影响面判定，在已确认的模式下建议具体走哪些阶段：

   ```
   📍 已确认模式：<模式>（路由声明时已选）
   📍 建议路径：<具体要走哪些阶段，用 → 连接>
      理由：<为什么走/跳过某些阶段>

   典型路径参考（Standard 模式）：
   - 完整：0-confirm → 1-analysis → 2-design → 3-task → 4-dev → 5-test → 6-review → 7-integration
   - 中等（需求已有增量）：1-analysis(增量) → 3-task → 4-dev → 5-test → 6-review → 7-integration
   - 跳设计（纯 bugfix）：3-task → 4-dev → 5-test → 6-review → 7-integration

   Fast 模式：短任务说明 → 修改 → 窄验证 → 结果证据
   Strict 模式：完整流程 + 安全/迁移/回滚检查 + 独立 review

   是否同意此路径？[Y/n]
   ```

## 输出

- `.specs/<req-id>/00-需求确认.md`（**必须严格按模板格式输出**）
- 一段路径建议（在已确认模式下建议具体阶段），并询问用户确认
- **更新 `.specs/项目状态.md`**：
  - `活跃 Req` 改为 `<req-id>`
  - `当前阶段` 设为 `confirm`
  - `阶段状态` 设为 `waiting_user_confirmation`
  - `上次完成阶段` 设为 `confirm`
  - `下一阶段` 设为建议路径中的下一阶段
  - `模式` / `判定理由` / `判定时间` 填入
  - 在「阶段进度」清单中打钩 `需求确认 → 00-需求确认.md`

## 约束（来自 RULES.md）

- 不允许跳过反问直接出方案
- 不允许凭空假设未确认的需求点
- 不允许在 00-需求确认.md 里写实现细节（那是 设计 的事）

## 自检（产出前自查）

- [ ] 00-需求确认.md 包含：Why / What / 影响面 / 范围排除 / 验收线
- [ ] 至少明确写出 1 条「本次不做」
- [ ] 路径建议有理由，不是默认全跑
- [ ] 没有跳到实现层（"用 Redux 存主题"这种话不该出现在这里）
- [ ] **步骤 2 已跑**：未命中就跳过；命中且用户选 2 → 00-需求确认.md 末尾有「架构层影响声明」段
- [ ] **项目状态.md 已更新**（活跃 Req + 当前阶段 + 模式 + 阶段进度打钩）

## 阶段完成声明（必须输出）

```
✅ 需求确认 完成
📝 产物：.specs/<req-id>/00-需求确认.md
📊 项目状态.md 阶段进度已更新：[x] 需求确认 → 00-需求确认.md

请确认需求确认书内容：
- Why: <一句话背景>
- What: <一句话目标>
- 影响面: <列出影响范围>
- 本次不做: <列出排除项>

确认后回复 Y 继续，或指出需要修改的地方。
```

> ⚠️ **关键阶段**：需求确认是关键阶段，必须等用户回复后才能进入下一阶段。

## 触发下一步

00-需求确认.md 经用户确认后，根据路径建议进入：
- `@devflow-kit/flow/prompts/1-analysis.md`（影响需求时）
- `@devflow-kit/flow/prompts/3-task.md`（直接拆任务时）
