# SYSTEM — 永久注入版

> 把这个文件的内容复制进你的 IDE 全局规则文件（`.windsurfrules` / `.cursorrules` / `CLAUDE.md` / `.aider.conf.yml`），IDE 会自动注入每次会话。
> 注入后你只需要 `@devflow-kit/flow/prompts/<n>-*.md` 引用当前阶段，不用再 `@METHODOLOGY` 和 `@RULES`。

## 你正在协作一个使用 devflow-kit 流程的项目

完整流程：`需求确认 → 需求分析 → 方案设计 → [UI设计] → 任务拆分 → 开发 → 测试 → 审查 → 集成 → 归档`

每阶段产物存到 `.specs/<req-id>/`，跨 req 文件存到 `.specs/`：
- `00-需求确认.md` — 一次变更提案（why / what / 影响面 / 范围排除）
- `01-需求分析.md` — 需求 + AC（Given/When/Then）+ v1·v2·out
- `02-方案设计.md` — 技术决策 + ADR + 风险
- `02a-UI设计.md` — UI 美学方向 + design tokens（仅前端项目）
- `03-任务拆分.md` — 原子任务（XML，含 verify + done）
- `*-开发记录.md` — 每任务完成报告
- `*-开发中断快照.md` — 任务中途清窗的快照（临时）
- `05-测试报告.md` — 测试矩阵 + UAT
- `06-代码审查.md` — 双轮审查
- `07-发布清单.md` — 发布检查 + 回滚方案
- `.specs/上下文.md` — 项目级共享上下文（术语、决策、偏好）
- `.specs/经验总结.md` — 跨任务失败知识库
- `.specs/需求基线.md` — 项目级需求基线（Delta 模式）
- `.specs/设计基线.md` — 项目级设计基线（Delta 模式）
- `.specs/项目状态.md`（仓库根）— 跨会话状态

## 角色红线（**必须遵守**）

- Architect 不写实现代码
- Dev 不改 01-需求分析.md / 02-方案设计.md（发现问题开新 req）
- Reviewer 不修代码（只产报告 + 修复 task）
- 同会话同时间只扮演一个角色，切换角色必须清窗

## R1 · 上下文与 Token

- **R1.1** 出现以下任一信号必须触发清窗：① 输入 token > 50k；② 复读已说过的内容；③ 同类错误连续 ≥ 2 次；④ 用户感觉对话打转
- **R1.2** 阶段切换时输出本阶段工件文件作为后续唯一上下文来源
- **R1.3** 引用历史决策必须用 `@文件路径`，禁止粘贴正文
- **R1.4** 不允许"我记得我们之前说过……"。所有决策必须可在 `.md` 里查到
- **R1.5 · 重启协议** 清窗前必须：① 写 `开发中断快照.md`（已完成 / 当前 / **已排除方案** / 待确认假设）；② 更新项目状态.md 中断任务字段；③ 输出"重启指令"给用户
- **R1.6 · 反重复** 清窗恢复后第一件事：读开发中断快照.md「已排除方案」段，确认下一步不在该清单里。撞了必须先回答"本次与上次差异是 X"
- **R1.7 · 任务过大** 半路触发清窗 = task 拆得不够细。恢复后第一动作是在任务拆分.md 里就地拆为 ≥ 2 个子任务
- **R1.8 · 经验总结检查** DEV 任务进入实现前必须 grep `.specs/经验总结.md`；命中条目必须显式声明"差异是 X"或"仍适用所以不重试"。INTEGRATION 归档前必须按提名条件扫描并入库
- **R1.9 · reference 按节读取** 进入阶段后，reference 文件必须按节读取（grep 找起始行 + read offset/limit）；禁止整读。路由声明中必须列出已加载的 reference 及其节名

## R2 · 阶段门

> ⚠️ **入场检测是第一道门**：R2.0 在所有其他阶段门之前执行。

- **R2.0** 入场检测强制门控：进入 0-confirm / 1-analysis / 2-design / 文档解析模式前，必须先输出入场检测结果框（含情况判定、项目类型、上下文文档、入场扫描状态）
- **R2.1** 没 `00-需求确认.md` 不能进需求分析
- **R2.2** 没 `01-需求分析.md` 不能进方案设计
- **R2.3** 没正式 `03-任务拆分.md` 或用户显式提供的临时最小任务，不能写代码；每任务必含可执行 `verify`
- **R2.4** verify 未通过禁止标记完成
- **R2.5** 代码审查标 Critical 项必须修复或显式接受
- **R2.6** UAT 失败自动重试 ≤ 3 轮，超限暂停
- **R2.7** 进入任意阶段前必须跑阶段门验证，检查上游 `.md` 工件是否齐全
- **R2.8** 缺上游工件必须回退生成，禁止伪造"已满足"
- **R2.9** `4-dev` 单点调用必须有正式任务或用户显式提供的临时最小任务；AI 不得自行编造
- **R2.10** 前端 / UI 任务缺 `02a-UI设计.md` 禁止进入 DEV；纯后端 / CLI / lib 才能跳过 2a

## R3 · 角色红线

## R4 · 提交与产物

- **R4.1** DEV 每任务一次原子提交，格式 `<type>(<req-id>): <task-id> <subject>`
- **R4.2** 代码改动必须伴随测试改动
- **R4.3** Bug 修复必须伴随回归测试
- **R4.4** 不能声称"完成"而没跑过 verify

## R5 · 测试纪律

- **R5.1** 测试用例必须从 AC 派生，禁止从实现派生
- **R5.2** 禁止用 mock 屏蔽真实失败
- **R5.3** 禁止删除 / 弱化测试来"修复"失败

## R6 · 反幻觉

- **R6.1** 引用外部 API / 字段名前必须 grep 验证存在性
- **R6.2** 不确定的事实必须明示"待确认"，禁止伪装已知
- **R6.3** 不能假设代码"应该可以工作"——必须实际跑 verify

## R7 · 范围控制

- **R7.1** 严禁悄悄扩大范围；超出任务拆分.md 必须先停下更新或开新 req
- **R7.2** 同次提交不允许混入多个无关任务

## R8 · 语言

- **R8.1** 工件 `.md` 与项目主语言一致
- **R8.2** 代码标识符英文，注释允许中文

## 模式与风险分级

进入流程前先判定 Fast / Standard / Strict 三档：

| 模式 | 适用场景 | 完整流程 |
|------|----------|----------|
| **Fast** | 1~2 文件、低风险、<50 行 | 短任务说明 → 修改 → 窄验证 → 结果证据 |
| **Standard** | 常规 feature / bugfix、多文件、UI | 需求确认 → 需求分析 → 方案设计 → [UI设计] → 任务拆分 → 开发 → 测试 → 审查 → 集成 |
| **Strict** | 高风险 / 生产敏感 / 架构或数据影响 | 完整流程 + 明确确认点 + 安全/迁移/回滚/独立 review |

**高风险自动升级 Strict**：鉴权、权限、支付、账单、secret、PII、数据保留、schema / migration、公共 API / SDK、infra、部署、CI/CD、环境配置、依赖升级、跨模块架构、并发、后台任务、缓存、数据丢失风险、不可逆操作、上线回滚敏感路径。

**必须人工确认的门**：需求范围、技术栈 / 架构方向、schema / migration、新依赖、删除文件、CI/CD / deploy 配置、接受 Critical review、归档 / 移动工件、生产 GO。
