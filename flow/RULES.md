# RULES — devflow-kit 流程规则（AI 内部文件）

> AI 指令：以下规则在运行 devflow-kit 的任意阶段均须遵守，**不读本文件直接运行即违反**。

## R1 · 状态管理与恢复

- **R1.1** 每轮输出前必须检查 `.specs/进度跟踪.md` 的 `活跃 req` 和 `当前阶段` 字段
- **R1.2** 跨阶段推进时必须先检查 `.specs/进度跟踪.md`，确认当前阶段是无误的。如果发现不一致，以 `.specs/进度跟踪.md` 为准，在回复开头报告差异并更新
- **R1.3** 换 req / 回旧阶段 / 跳阶段前必须先读 `.specs/进度跟踪.md` 并比对会话上下文；不一致 → 优先信任项目状态文件的内容
- **R1.4** 每次阶段切换（含同一 req 的跨阶段推进）必须先更新 `.specs/进度跟踪.md` 再进入新阶段
- **R1.5** 中断重启协议：
  1. 用户说"继续"或"下一阶段" → 先读 `.specs/进度跟踪.md`
  2. 对到 `活跃 req` + `当前阶段`
  3. 加载对应的 prompt → 按 `PROGRESS` 内容继续执行
  4. AI 必须输出一段重启摘要：我的记忆来自哪里（最近一份成果 / 日志），而不是假装从未中断
- **R1.6** 反重复检查：恢复中断任务时，必须读 `<task-id>-开发中断快照.md` 的「已排除方案」段，确认下一步不与已排除方案撞车
- **R1.7** 任务过大拆分：若执行中发现 task 体量明显过大（单 task 改动 > 10 文件 / > 500 行），应在 `03-任务拆分.md` 里就地拆为子任务后再继续
- **R1.8** 经验扫描：每个 DEV 任务开工前必须扫 `.specs/经验总结.md`，检查是否有可复用经验；INTEGRATION 阶段必须提名新条目
- **R1.9** Token 预算控制：进入任何阶段的首轮消息，加载的 reference/* 总行数 ≤ 150 行。超过 → 拆到后面按需拉

## R2 · 重新执行协议

- **R2.1** AI 判断错误或偏离需求时，用户说"重做"、"重新来"、"失败"、"不对" → **先暂停当前工作**，然后主动提出回溯方案——不是直接回退到某个阶段，而是问用户"是回退到上一阶段还是回到需求"
- **R2.2** 如果 phase‑1 已经完成但内容不完整或质量不达标，同一个 phase 需要重新执行时（仍然产出新文件），必须遵守以下规则：
  1. 归档：旧的 `.specs/<id>/<n>-*.md` 应当重命名为 `.specs/archive/<id>-<n>-*.md`
  2. 重做：在 `.specs/<id>/` 下重新生成
  3. 记录：在归档说明中标注原因和覆盖范围
- **R2.3** 任何时候感到迷茫、跨度过大、前一步产出不适合当前阶段 → 暂停并向用户描述困境，给出 2~3 个具体选项
- **R2.4** verify 未通过禁止标记任务完成。必须实际跑过验证命令并贴出真实输出
- **R2.5** 所有 Critical 审查发现必须修复或显式「已知接受」并经人工确认，否则禁止进 INTEGRATION
- **R2.6** 自动重试上限：任何阶段（plan / dev / verify / UAT）的自动重试次数 ≤ 3 轮。超限必须停下来让人决策，禁止死循环

## R3 · 产物完整性

- **R3.1** 产物必须存放在 `.specs/<req-id>/` 下，确保跨会话可查找
- **R3.2** 产物文件名统一用 `<n>-<英文名称>.md` 格式（如 `02-方案设计.md`），禁止中文文件名
- **R3.3** 跨阶段推进时，新的产物 `/n/` → `../../` 引用旧产物，不要复制粘贴
- **R3.4** 产物文件长度 ≤ 500 行（多数 < 200 行），超过 → 拆 `-a.md` / `-b.md`
- **R3.5** `.specs/上下文.md` 是通用域语言文件，存放术语表、已锁设计决策、全局约束
- **R3.6** `.specs/经验总结.md` 是通用经验沉淀文件，存放跨 req 的可复用经验
- **R3.7** 所有产物必须有明确的 **Req-ID 命名空间**，多需求并行时使用 `.specs/<id>/` 隔离
- **R3.8** `.specs/进度跟踪.md` 中记录决策日志（最近 10 条倒序），每条格式 `[YYYY-MM-DD] 内容 — 来源`

## R4 · 交互节奏

- **R4.1** 每个用户回合结束时，必须输出**当前阶段进展** + **下一步任务**（不要留 "还有什么要帮忙的吗"）
- **R4.2** 用户打断 / 换话题时 → 先记录当前进展到 `.specs/进度跟踪.md` 的 `中断任务` 字段，再处理新话题
- **R4.3** 每个任务一个 fresh context；不允许把多个任务塞进同一个会话
- **R4.4** 禁止声称"通过"而没贴真实输出（测试、构建、lint 等都必须贴真实结果）
- **R4.5** Schema 变更必伴随迁移文件（含 up + down）。只改 model 不生迁移就提交 → 违规，AI 必须回滚
- **R4.6** 破坏性变更（删 ≥ 5 行 / 改公共接口）必走完整协议：grep 引用图 + 反问用户 + 回归测试覆盖

## R5 · 自动推进

- **R5.1** 用户以 `"继续"` / `"继续下一阶段"` / `"下一阶段"` / `"y"` / `"Y"` / `"yes"` 回复时，AI 自动按当前阶段的「触发下一步」路由进入下一阶段。不反问"确定吗"，不重新确认需求。
- **R5.2** 如果当前阶段的响应末尾包含了 `[Y/n]` 提示，用户回车或回复 `y` / `Y` / `yes` 等同确认推进。AI 直接打开下一阶段的 prompt 继续。
- **R5.3** 中断条件是：用户回复中包含 `"n"` / `"N"` / `"no"` / `"等等"` / `"有问题"` / `"先不"` 等否定词。此时 AI 暂停推进，追问用户意图。
- **R5.4** 禁止用 mock 屏蔽真实失败。测试失败必须修实现或修测试逻辑，不允许通过 mock 绕过

## R6 · 命名与标识

- **R6.1** 所有 ID（Req-ID、Task-ID、文件名）只由英文小写字母 + 连字符组成
- **R6.2** Req-ID 格式：`<一句话标识>`，例如 `add-search-box`、`fix-login-redirect`
- **R6.3** Task-ID 格式：`T<两位数字>`，例如 `T01`、`T12`
- **R6.4** 路由声明中出现多阶段标识时使用英文标识（`0-confirm → 1-analysis`），不加中文注释，不混用
- **R6.5** 提交前必跑 diff 边界 verify：`git diff --name-only` 检查是否所有改动都在当前 task `write_files` 范围内。越界必需回滚或扩范围（需人工同意）

## R7 · 安全与范围

- **R7.1** AI 不能替用户做升级决定。高风险升级 Strict 必须人工确认
- **R7.2** 所有「必须人工确认的门」列表中列出的操作必须等用户确认
- **R7.3** 不可执行不可逆操作（删除文件、修改 deploy 配置等）除非用户显式授权
- **R7.4** 所有产物的写入必须先经 checklist 展示内容概览后再实施

## R8 · 模式判定与降级

- **R8.1** 阶段结束后 AI 必须评估是否需要降级。判定表见 GO.md「模式降级评估」
- **R8.2** 降级必须显式声明（含触发信号和裁剪内容），不允许悄然跳过阶段
- **R8.3** 降级前必须暂停并询问用户是否同意，用户确认后才可执行降级；升级同理必须暂停并反问用户。AI 不得自行决定降级。

## R9 · 中文/英文

- **R9.1** 工件文件（`*.md`）的语言保持项目当前主语言一致。
- **R9.2** 代码标识符（变量名、函数名）一律英文，注释允许中文。

## R10 · UI 设计约束

- **R10.1** 颜色必须使用 OKLCH（除非有遗留约束并显式声明例外理由）
- **R10.2** 字体必须避开 AI slop 默认集（Inter / Roboto / Arial / Helvetica / system-ui）；用了必须解释为什么不能换
- **R10.3** 每个美学决策必须给理由（"为什么选这个调性 / 这个 hue / 这个字体"），不允许"看起来比较好"
- **R10.4** 02a-UI设计.md 输出前必须逐条对照 `ui-anti-patterns.md` 完成自检，命中的必须改或显式标注例外理由
- **R10.5** brownfield 项目必走视觉语汇对齐（2a-ui-design 步骤 1.5），观察报告必须经用户确认后才能进步骤 2
- **R10.6** v0 草稿必须在写完整 02a-UI设计.md 前给用户确认（除非是极小改动且用户明说"别 v0 直接写"）
- **R10.7** 禁止编造数据 / 好评 / logo；缺素材走占位符策略（见 2a-ui-design 步骤 7）

## R11 · 测试纪律

- **R11.1** 测试用例从 AC 派生，不从实现派生
- **R11.2** bug 修复必伴随回归测试，加入 `.specs/经验总结.md`
- **R11.3** 禁止通过删除 / 弱化测试来"修复"失败
- **R11.4** 不允许跳过任何测试轮次而没有显式理由（在范围声明里写明）
- **R11.5** 性能 / 安全 / 兼容轮次的"通过 / 失败"必须基于**可量化指标**或工具输出，不允许"看起来没问题"
