# RULES — devflow-kit 流程规则（AI 内部文件）

> AI 指令：以下规则在运行 devflow-kit 的任意阶段均须遵守，**不读本文件直接运行即违反**。

---

## R1 · 状态管理与恢复

- **R1.1** 以下情况必须读取 `.specs/项目状态.md`：进入 Standard / Strict、用户说"继续 / 下一阶段 / 执行 Txx"、阶段切换、会话恢复、发现上下文与产物不一致。
- **R1.2** `.specs/项目状态.md` 使用明确状态机：`当前阶段` 表示正在处理或等待确认的阶段；`阶段状态` 表示 `not_started / in_progress / waiting_user_confirmation / completed / blocked / skipped`；`上次完成阶段` 和 `下一阶段` 用于恢复。
- **R1.3** 换 req / 回旧阶段 / 跳阶段前必须先读 `.specs/项目状态.md` 并比对会话上下文；不一致 → 优先信任项目状态文件的内容
- **R1.4** 每次阶段切换必须先更新 `.specs/项目状态.md`：完成当前阶段、设置阶段状态、写入上次完成阶段和下一阶段，再进入新阶段。
- **R1.5** 中断重启协议（会话压缩/清窗后必须执行）：
  1. 用户说"继续"或"下一阶段" → 先读 `.specs/项目状态.md`
  2. 对到 `活跃 req` + `当前阶段` + `当前 Task`
  3. **按项目状态.md「🔄 压缩恢复指引」执行恢复步骤**
  4. 读当前阶段的前置产物文件（见项目状态.md「恢复时必须加载的文件」表）
  5. AI 必须输出恢复摘要（格式见项目状态.md），明确说明记忆来自哪里
  6. 检查「阶段进度」清单，确认哪些已完成、哪些待做
  7. 从最后一个已完成阶段的下一步继续，不跳过、不重做已完成的
- **R1.6** 反重复检查：恢复中断任务时，必须读 `<task-id>-开发中断快照.md` 的「已排除方案」段，确认下一步不与已排除方案撞车
- **R1.7** 任务过大拆分：若执行中发现 task 体量明显过大（单 task 改动 > 10 文件 / > 500 行），应在 `03-任务拆分.md` 里就地拆为子任务后再继续
- **R1.8** 经验扫描：每个 DEV 任务开工前必须扫 `.specs/经验总结.md`，检查是否有可复用经验；INTEGRATION 阶段必须提名新条目
- **R1.9** Token 预算控制：进入任何阶段的首轮消息，加载的 reference/* 总行数 ≤ 150 行。超过 → 拆到后面按需拉
- **R1.10** 产物完整性检查：恢复后必须核对「阶段进度」清单，确认所有已打钩（`[√]`）阶段的产物文件都存在。缺失 → 报告异常，不继续
- **R1.11** 入场扫描强制门控：
  - 进入任何需求相关阶段（0-confirm / 1-analysis / 2-design / 文档解析模式）前，**必须先输出入场检测结果框**
  - 检测结果框必须包含：情况判定（A/B/C/D/E/F）、项目类型（brownfield/greenfield）、上下文文档路径、入场扫描状态
  - 未输出检测结果框 → 禁止进入后续阶段
  - brownfield 项目检测结果为"待用户确认"（情况 D/E）→ 禁止继续，必须等用户回复
  - 检测结果为"已跳过"时必须注明跳过原因（用户手动选择 / greenfield 项目）
- **R1.12** reference 按节读取强制：
  - 进入阶段后，必须按 GO.md 第四步「查表」列的要求按节读取 reference 文件
  - **禁止整读**：tech-stacks.md / test-pyramid.md / ui-aesthetics.md / 4-dev-rules.md / frontend-engineer-rules.md 必须用 grep 找节起始行 + read offset/limit 读取
  - **唯一例外**：ui-anti-patterns.md（仅 111 行）可全读
  - 路由声明中必须列出已加载的 reference 及其节名和行数
  - 未在路由声明中列出 → 视为未加载，必须补做

## R2 · 阶段门

> ⚠️ **入场检测是第一道门**：R2.0 在所有其他阶段门之前执行。未完成入场检测 → 禁止进入任何需求相关阶段。

- **R2.0** 入场检测强制门控（见 R1.11）：进入 0-confirm / 1-analysis / 2-design / 文档解析模式前，必须先输出入场检测结果框
- **R2.1** 没 `00-需求确认.md` 不能进需求分析
- **R2.2** 没 `01-需求分析.md` 不能进方案设计
- **R2.3** 没正式 `03-任务拆分.md` 或用户显式提供的临时最小任务，不能写代码；每任务必含可执行 `verify`
- **R2.4** verify 未通过禁止标记完成
- **R2.5** 代码审查标 Critical 项必须修复或显式接受
- **R2.6** UAT 失败自动重试 ≤ 3 轮，超限暂停
- **R2.7** 进入任意阶段前必须跑阶段门验证，检查上游 `.md` 工件是否齐全
- **R2.8** 缺上游工件必须回退生成，禁止伪造"已满足"
- **R2.9** `4-dev` 单点调用必须有正式任务或用户显式提供的临时最小任务；AI 不得自行编造
- **R2.10** 前端 / UI 任务缺 `02a-UI设计.md` 禁止进入 DEV；纯后端 / CLI / lib 才能跳过 2a

## R3 · 角色红线

- **R3.1** Architect 不写实现代码
- **R3.2** Dev 不改 01-需求分析.md / 02-方案设计.md（发现问题开新 req）
- **R3.3** Reviewer 不修代码（只产报告 + 修复 task）
- **R3.4** 同会话同时间只扮演一个角色，切换角色必须清窗

## R4 · 提交与产物

- **R4.1** DEV 每任务一次原子提交，格式 `<type>(<req-id>): <task-id> <subject>`
- **R4.2** 代码改动必须伴随测试改动
- **R4.3** Bug 修复必须伴随回归测试
- **R4.4** 禁止声称"通过"而没贴真实输出（测试、构建、lint 等都必须贴真实结果）
- **R4.5** Schema 变更必伴随迁移文件（含 up + down）。只改 model 不生迁移即标记完成 → 违规，AI 必须暂停并修复或回滚本次改动
- **R4.6** 破坏性变更（删 ≥ 5 行 / 改公共接口）必走完整协议：grep 引用图 + 反问用户 + 回归测试覆盖

## R5 · 测试纪律

- **R5.1** 测试用例必须从 AC 派生，禁止从实现派生
- **R5.2** 禁止用 mock 屏蔽真实失败
- **R5.3** 禁止删除 / 弱化测试来"修复"失败
- **R5.4** 不允许跳过任何测试轮次而没有显式理由（在范围声明里写明）
- **R5.5** 性能 / 安全 / 兼容轮次的"通过 / 失败"必须基于**可量化指标**或工具输出，不允许"看起来没问题"

## R6 · 反幻觉

- **R6.1** 引用外部 API / 字段名前必须 grep 验证存在性
- **R6.2** 不确定的事实必须明示"待确认"，禁止伪装已知
- **R6.3** 不能假设代码"应该可以工作"——必须实际跑 verify
- **R6.4** 沿用既有抽象：写新代码前必须 grep 同类抽象（utils/ / helpers/ / lib/ / shared/ 等），找到已有实现就用，不另起炉灶
- **R6.5** 开发完成后必跑 diff 边界 verify：`git diff --name-only` 检查是否所有改动都在当前 task `write_files` 范围内。越界必需回滚或扩范围（需人工同意）

## R7 · 范围控制

- **R7.1** 严禁悄悄扩大范围；超出任务拆分.md 必须先停下更新或开新 req
- **R7.2** 同次提交不允许混入多个无关任务
- **R7.3** AI 不能替用户做升级决定。高风险升级 Strict 必须人工确认
- **R7.4** 不可执行不可逆操作（删除文件、修改 deploy 配置等）除非用户显式授权

## R8 · 自动推进与人工确认门

### R8.1 阶段分类（关键阶段 vs 执行阶段）

| 类型 | 阶段 | 确认规则 |
|------|------|----------|
| **关键阶段** | Standard/Strict 模式确认、需求确认、需求分析、方案设计、2a UI设计 | **必须等用户回复**，不能自动推进 |
| **执行阶段** | 任务拆分、开发、测试、审查、集成 | 如果当前回合仍有明确下一步、工具权限和无风险阻塞，可以继续；否则输出下一步建议 |

### R8.2 关键阶段确认规则（必须停等）

以下阶段完成后，AI **必须停下来等用户回复**，不能自动继续：

- **模式确认**：输出模式确认后，必须等用户选择
- **需求确认**：产出 00-需求确认.md 后，必须等用户确认内容
- **需求分析**：产出 01-需求分析.md 后，必须等用户确认 AC 和范围
- **方案设计**：产出 02-方案设计.md 后，必须等用户确认技术栈和关键决策
- **2a UI设计**：产出 02a-UI设计.md 后，必须等用户确认视觉方向

输出格式：
```
✅ <阶段> 完成
📝 产物：.specs/<req-id>/<产物文件名>

请确认产物内容是否正确：
- <列出关键内容点，让用户快速确认>

确认后回复 Y 继续，或指出需要修改的地方。
```

### R8.3 执行阶段推进规则

以下阶段完成后，AI 可以在无阻塞时继续推进：

- **任务拆分** → 开发
- **开发** → 测试
- **测试** → 审查
- **审查** → 集成

输出格式：
```
✅ <阶段> 完成
📝 产物：.specs/<req-id>/<产物文件名>
📊 项目状态.md 阶段进度已更新：[√] <阶段> → <产物文件名>
➡️ 下一步：<下一阶段或具体动作>
```

如果需要用户选择、工具权限不足、发现风险、出现失败或上下文不足，则必须暂停并说明最小决策点。

### R8.4 阶段完成后必须更新项目状态.md

**每个阶段完成后，AI 必须执行以下步骤**：

1. **产出产物文件** → 保存到 `.specs/<req-id>/`
2. **更新项目状态.md**：
   - `当前阶段` 保持为当前阶段或设为下一阶段（取决于是否已经实际进入）
   - `阶段状态` 设为 `completed` / `waiting_user_confirmation` / `blocked`
   - `上次完成阶段` 设为刚完成的阶段
   - `下一阶段` 设为建议下一步
   - 在「阶段进度」清单中，将当前阶段改为 `[√]`（已完成）
3. **输出完成声明**（含产物路径 + 项目状态更新确认 + 下一步）

**禁止行为**：
- ❌ 产出产物后不更新项目状态.md
- ❌ 进入下一阶段前不更新阶段进度打钩
- ❌ 假装已完成但项目状态.md 未更新

### R8.5 中断条件（必须停下来等用户）

遇到以下情况，AI **必须暂停**，不能自动推进：

- Critical 修复待处理
- 模式需要升级或降级
- 多路由方向需要用户选择
- 需人工确认的门（见 R7.3）
- 用户明确说"暂停" / "等等" / "有问题"

## R9 · 命名与标识

- **R9.1** 所有 ID（Req-ID、Task-ID、文件名）只由英文小写字母 + 连字符组成
- **R9.2** Req-ID 格式：`<一句话标识>`，例如 `add-search-box`、`fix-login-redirect`
- **R9.3** Task-ID 格式：`T<两位数字>`，例如 `T01`、`T12`
- **R9.4** 路由声明中出现多阶段标识时使用英文标识（`0-confirm → 1-analysis`），不加中文注释，不混用

## R10 · 模式判定与降级

- **R10.1** 阶段结束后 AI 必须评估是否需要降级。判定表见 GO.md「模式降级评估」
- **R10.2** 降级必须显式声明（含触发信号和裁剪内容），不允许悄然跳过阶段
- **R10.3** 降级前必须暂停并询问用户是否同意，用户确认后才可执行降级；升级同理必须暂停并反问用户。AI 不得自行决定降级。

## R11 · 中文/英文

- **R11.1** 工件文件（`*.md`）的语言保持项目当前主语言一致。
- **R11.2** 代码标识符（变量名、函数名）一律英文，注释允许中文。

## R12 · UI 设计约束

- **R12.1** 颜色必须使用 OKLCH（除非有遗留约束并显式声明例外理由）
- **R12.2** 字体必须避开 AI slop 默认集（Inter / Roboto / Arial / Helvetica / system-ui）；用了必须解释为什么不能换
- **R12.3** 每个美学决策必须给理由（"为什么选这个调性 / 这个 hue / 这个字体"），不允许"看起来比较好"
- **R12.4** 02a-UI设计.md 输出前必须逐条对照 `ui-anti-patterns.md` 完成自检，命中的必须改或显式标注例外理由
- **R12.5** brownfield 项目必走视觉语汇对齐（2a-ui-design 步骤 1.5），观察报告必须经用户确认后才能进步骤 2
- **R12.6** v0 草稿必须在写完整 02a-UI设计.md 前给用户确认（除非是极小改动且用户明说"别 v0 直接写"）
- **R12.7** 禁止编造数据 / 好评 / logo；缺素材走占位符策略（见 2a-ui-design 步骤 7）

## R13 · 产物完整性

- **R13.1** 产物必须存放在 `.specs/<req-id>/` 下，确保跨会话可查找
- **R13.2** 产物文件名使用项目当前中文模板命名（如 `00-需求确认.md`、`01-需求分析.md`、`02-方案设计.md`）。内部状态枚举可用英文，但产物文件名必须与模板和 README 保持一致。
- **R13.3** 跨阶段推进时，新的产物 `/n/` → `../../` 引用旧产物，不要复制粘贴
- **R13.4** 产物文件长度 ≤ 500 行（多数 < 200 行），超过 → 拆 `-a.md` / `-b.md`
- **R13.5** `.specs/上下文.md` 是通用域语言文件，存放术语表、已锁设计决策、全局约束
- **R13.6** `.specs/经验总结.md` 是通用经验沉淀文件，存放跨 req 的可复用经验
- **R13.7** 所有产物必须有明确的 **Req-ID 命名空间**，多需求并行时使用 `.specs/<id>/` 隔离
- **R13.8** `.specs/项目状态.md` 中记录决策日志（最近 10 条倒序），每条格式 `[YYYY-MM-DD] 内容 — 来源`
- **R13.9** **⚠️ 模板强制**：输出任何产物前必须先读取对应模板文件（`flow/templates/*.md`），产物必须包含模板所有段落，不得省略或改写结构。占位符（如 `<req-id>`）必须替换为实际值。
- **R13.10** **模板校验失败处理**：若发现产物遗漏模板段落，必须立即补齐后再标记完成。禁止声称"已完成"但产物不完整。

---

## 违规处理

检测到自己即将违反任一规则时，先输出：
` 规则 R{编号} 触发：<原因>。需要人工决策。`
然后停下等待，**禁止"自我授权"绕开规则**。

---

## AI 常见违规行为速查

| 违规行为 | 触发规则 | 正确做法 |
|----------|----------|----------|
| 未输出入场检测结果框就进入路由 | R1.11 / R2.0 | 先输出检测结果框 |
| 情况 D/E 自行跳过 | R1.11 | 停等用户确认 |
| 未加载 skill 就输出路由声明 | 第五门控 | 先加载所有 skill |
| 未读模板就输出产物 | R13.9 | 先读取模板文件 |
| 产物遗漏模板段落 | R13.9 | 包含所有必填段落 |
| verify 未通过就标记完成 | R2.4 | verify 必须通过 |
| 整读 reference 文件 | R1.12 | 按节读取 |
| 悄悄扩大范围 | R7.1 | 先停下更新任务拆分 |
| 自行决定模式降级 | R10.3 | 询问用户确认 |
