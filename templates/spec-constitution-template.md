# 项目宪法 (Project Constitution)

> **来源**: 融合 spec-kit 的 Constitutional Foundation
> **用途**: 建立项目的不可变原则，约束所有后续设计和实现决策

---

## 核心原则

### I. 库优先 (Library-First)

每个功能**必须**先以独立库的形式存在，然后才能集成到应用代码中。

**要求**：
- 库必须是自包含的、可独立测试的、有文档的
- 必须有明确的用途——不允许"组织用途"式的库
- 库的接口必须是清晰的，不依赖特定框架的实现细节

### II. CLI 接口 (CLI Interface)

每个库**必须**通过命令行接口暴露其功能。

**要求**：
- 输入：stdin / 参数 / 文件（文本形式）
- 输出：stdout（文本形式），错误 → stderr
- 必须支持 JSON 格式用于结构化数据交换
- 确保可观测性和可测试性

### III. 测试先行 (Test-First) — **不可协商**

**严禁**在测试之前编写实现代码。

**流程**：
1. 先写单元测试
2. 用户审核并批准测试
3. 确认测试失败（Red 阶段）
4. 然后才编写实现

**红线**：
- 不允许"先写代码再补测试"
- 不允许为了通过测试而弱化测试
- 不允许跳过测试阶段而没有明确理由

### IV. 集成测试优先 (Integration-First Testing)

优先使用真实环境测试，而非隔离的 mock。

**要求**：
- 优先使用真实数据库而非 mock
- 优先使用真实服务实例而非 stub
- 实现前必须有契约测试
- 关注：新库契约测试、契约变更、服务间通信、共享 Schema

### V. 可观测性 (Observability)

所有服务**必须**可观测。

**要求**：
- 文本 I/O 确保可调试性
- 必须有结构化日志
- 关键指标必须可采集（错误率、响应时间、吞吐量）

### VI. 版本与破坏性变更 (Versioning & Breaking Changes)

**格式**：MAJOR.MINOR.BUILD

**规则**：
- MAJOR：破坏性变更
- MINOR：新增功能，向后兼容
- BUILD：Bug 修复，向后兼容

### VII. 简单性 (Simplicity)

**约束**：
- 初始实现最多 3 个项目
- 超过 3 个项目需要文档化理由
- 禁止"将来可能需要"式的过度设计

### VIII. 反抽象 (Anti-Abstraction)

**规则**：
- 直接使用框架功能，而非包装它们
- 单一模型表示，避免多层抽象
- 只有在明确需要跨框架移植时才创建抽象层

### IX. 宪法优先级

本宪法**优先于**所有其他实践和约定。

**修订流程**：
- 必须有文档化的修改理由
- 必须经过项目维护者审核批准
- 必须评估向后兼容性
- 必须有迁移计划
