## 模板强制质量机制

### 1. 防止过早实现细节

模板明确指导：
- ✅ 关注用户需要**什么**和**为什么**
- ❌ 避免**如何**实现（不含技术栈、API、代码结构）

### 2. 强制显式不确定性标记

创建规格时：
1. **标记所有歧义**：使用 `[NEEDS CLARIFICATION: 具体问题]`
2. **不猜测**：如果提示未指定某事，标记它

示例：
- `[NEEDS CLARIFICATION: 认证方式未指定 - 邮箱密码/SSO/OAuth?]`
- `[NEEDS CLARIFICATION: 保留期限未指定]`

### 3. 通过清单进行结构化思考

模板包含全面清单，作为规格的"单元测试"：

```markdown
### 需求完整性

- [ ] 无 [NEEDS CLARIFICATION] 标记残留
- [ ] 需求可测试且无歧义
- [ ] 成功标准可度量
```

### 4. 通过门禁实现宪法合规

实现计划模板通过阶段门禁强制架构原则：

```markdown
### Phase -1: 实现前门禁

#### 简单性门禁 (Article VII)

- [ ] 使用 ≤3 个项目？
- [ ] 没有过度设计？
```

## 测试先行思维

实现模板强制测试先行开发：

```markdown
### 文件创建顺序
1. 创建 `contracts/` 及 API 规格
2. 按顺序创建测试文件：契约 → 集成 → e2e → 单元
3. 创建源文件使测试通过
```

此顺序约束确保 AI 在实现前思考可测试性和契约。

---

## 防止投机性功能

模板明确阻止投机：

```markdown
- [ ] 无投机性或"可能需要"的功能
- [ ] 所有阶段有明确的前置条件和交付物
```

这阻止 AI 添加增加实现复杂性的"锦上添花"功能。
