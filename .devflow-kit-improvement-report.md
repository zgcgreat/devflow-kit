## 🔴 P0 - 阻塞性问题（必须修复）

### 问题 1: STATE.md 更新时机不一致

**现象**：
- 在 caveman 项目中，AI 有时在阶段完成前就更新了 STATE.md
- 有时在阶段完成后忘记更新 STATE.md
- 用户手动回退时发现 STATE.md 与实际进度不匹配

**影响**：
- 中断恢复时可能读取到错误的阶段状态
- 导致 AI 误判当前进度，跳过必要步骤

**根本原因**：
- GO.md Step 7 要求"执行 Stage Skill → 更新 STATE.md"
- 但各 Stage Skill 的自检清单中，STATE.md 更新的位置不统一
- 有些在产物生成后立即更新，有些在最后才更新

**建议修复**：

1. **在 RULES.md 中增加 R8.6**：
   ```markdown
   ## R8.6 STATE.md 更新时机强制规则
   
   **必须在以下时刻更新 STATE.md**：
   1. 阶段产物生成并验证通过后（立即更新，不得延迟）
   2. 进入下一阶段前（确认上一阶段已完成）
   3. 会话中断前（保存当前进度）
   
   **禁止行为**：
   - ❌ 产物未验证通过就更新 STATE.md
   - ❌ 声称"阶段完成"但 STATE.md 未更新
   - ❌ 跨阶段推进时不更新阶段进度打钩
   
   **更新顺序**：
   1. 生成产物文件
   2. 验证产物完整性（模板段落核对）
   3. 更新 STATE.md（当前阶段、阶段状态、上次完成阶段、下一阶段）
   4. 输出完成声明（含产物路径 + STATE.md 更新确认）
   ```

2. **在所有 Stage Skill 的自检清单中增加**：
   ```markdown
   - ⏳ STATE.md 已更新（当前阶段、阶段状态、上次完成阶段、下一阶段）
   - ⏳ 阶段进度清单已打钩（[x] <阶段名称>）
   ```

3. **在 GO.md Step 7 中明确**：
   ```markdown
   ## Step 7 · 执行 Stage Skill
   
   Stage Skill 按**其内部定义的 Step 流程**执行：
   1. 入口门禁检查
   2. 执行 Step 1-N
   3. 产物输出前 read_file 模板
   4. 执行自检清单
   5. **⚠️ 更新 STATE.md（必须在输出完成声明前）**
   6. 输出完成声明（含 STATE.md 更新确认）
   7. 路由到下一阶段
   ```

### 问题 2: 模板读取强制执行点不够明确

**现象**：
- 在 caveman 项目中，AI 有时会"凭记忆"生成产物，未实际读取模板
- 导致遗漏某些必填段落（如 02-design.md 的"既有架构对齐"章节）

**影响**：
- 产物不完整，违反 R13.9
- 弱 AI 容易遗漏关键段落

**根本原因**：
- 虽然 RULES.md R13.9 和 Stage Skill 都要求 read_file 模板
- 但缺乏**强制执行点**（即 AI 必须在哪个具体步骤读取）
- 没有明确的"如果不读取会怎样"的后果说明

**建议修复**：

1. **在 RULES.md 中增加 R13.12**：
   ```markdown
   ## R13.12 模板读取强制执行点
   
   **每个 Stage 必须在输出产物前的最后一个 Step 执行**：
   
   ```python
   # 伪代码示例（必须出现在 Stage Skill 的 Step N-1）
   print("⚠️ 强制检查点：即将输出产物")
   print("正在读取模板文件...")
   read_file("flow/templates/<template-name>.md")
   print("✅ 模板已读取，开始逐项核对必填段落")
   ```
   
   **如果 AI 未执行 read_file 就输出产物**：
   - 视为严重违规，必须重新生成产物
   - 在产物开头添加警告注释：`<!-- ⚠️ 此产物未经过模板核对，可能存在遗漏 -->`
   
   **推荐做法**：
   - 在 Stage Skill 的最后一步之前，插入"模板读取与核对"Step
   - 该 Step 必须包含：
     1. read_file 调用
     2. 提取必填段落清单
     3. 逐项核对（输出表格）
     4. 如有缺失，立即补齐
   ```

2. **在所有 Stage Skill 中增加统一的"模板核对 Step"**：
   ```markdown
   ### Step N-1: 模板读取与核对（强制执行点）
   
   **⚠️ 这是输出产物前的最后检查点，不得跳过**
   
   1. 读取模板文件：
      ```python
      read_file("flow/templates/<template-name>.md")
      ```
   
   2. 提取必填段落清单（从模板开头的「必填段落清单」章节）
   
   3. 逐项核对：
      ```markdown
      □ 段落 A → 已包含？✅/❌
      □ 段落 B → 已包含？✅/❌
      ...
      ```
   
   4. 如有缺失（任一 ❌）：
      - **禁止输出产物**
      - 立即补齐缺失段落
      - 重新核对直到全部 ✅
   
   5. 全部 ✅ 后，进入 Step N（输出产物）
   ```

- AI 可能无法正确执行恢复流程

**建议修复**：

1. **在 `docs/tutorials/` 下新增 `recovery-demo.md`**：
   ```markdown
   # 中断恢复演练
   
   ## 场景模拟
   
   ### 步骤 1: 启动一个需求
   ```
   Use devflow-kit.
   
   我要做一个用户登录功能
   ```
   
   AI 执行到 2-design 阶段，生成了 02-design.md
   
   ### 步骤 2: 模拟中断
   - 关闭对话窗口
   - 或清空上下文
   - 或说"我们换个话题"
   
   ### 步骤 3: 恢复中断
   ```
   Use devflow-kit. 继续
   ```

   1. 读取 `.devflow-kit/STATE.md`
   2. 发现 `活跃 req: user-login`，`当前阶段: design`，`阶段状态: completed`
   3. 输出恢复摘要：

      - 上次完成阶段：design（02-design.md 已生成）

      - ✅ 00-requirements.md
      - ✅ 01-analysis.md
      - ✅ 02-design.md
      
      准备进入 task 阶段...
      ```
   4. 继续执行 stage-3-task
   
   ## 常见问题
   
   ### Q1: STATE.md 与实际进度不一致怎么办？
   A: 优先信任 STATE.md，但如果发现明显错误（如产物文件不存在），应报告异常并请求用户确认。
   
   ### Q2: 多个中断任务怎么处理？

   ```

2. **在 README.md 的"使用场景"部分增加**：

   ### 场景 4: 中断恢复演练

### 问题 5: 缺少"模式切换"的实际案例

**现象**：
- GO.md 和 RULES.md 都提到了模式降级/升级
- 但没有实际案例说明何时应该切换模式
- 用户在什么情况下应该从 Standard 切换到 Strict？

**影响**：
- 用户不清楚模式选择的实际应用场景
- AI 可能推荐不合适的模式

**建议修复**：

1. **在 `docs/modes.md` 中增加"模式切换案例"章节**：
   ```markdown
   ## 模式切换案例
   
   ### 案例 1: Standard → Strict（风险升级）
   
   **场景**：原本计划修改前端文案（Standard），但发现需要同步修改后端 API
   
   **触发信号**：
   - 改动范围超出预期（从 2 文件 → 5 文件）
   - 涉及数据迁移（数据库 schema 变更）
   - 影响生产环境（API 接口变更）
   
   **AI 输出**：
   ```markdown
   ⚠️ 模式升级建议
   
   原计划：Standard 模式（前端文案修改）
   实际情况：涉及后端 API 和数据库变更
   
   建议升级到 Strict 模式，理由：
   1. 改动范围扩大（2 → 5 文件）
   2. 涉及数据迁移（高风险）
   3. 影响生产 API（需回归测试）
   
   Strict 模式额外流程：
   - 增加架构评审（stage-a-architect）
   - 完整回归测试套件
   - 灰度发布计划
   
   是否同意升级？回复 Y/N
   ```
   
   ### 案例 2: Strict → Standard（风险降级）
   
   **场景**：原本担心有风险的数据库查询优化，经分析发现只是加索引
   
   **触发信号**：
   - 风险评估降低（从"数据迁移"→"只读操作"）
   - 改动范围缩小（从 5 文件 → 2 文件）
   - 无破坏性变更
   
   **AI 输出**：
   ```markdown
   ℹ️ 模式降级建议
   
   原计划：Strict 模式（担心数据迁移风险）
   实际情况：仅添加数据库索引，无 schema 变更
   
   建议降级到 Standard 模式，理由：
   1. 无破坏性变更（只读操作）
   2. 改动范围小（2 文件）
   3. 可快速回滚（DROP INDEX）
   
   是否同意降级？回复 Y/N
   ```
   ```

2. **在 GO.md 的"模式判定"部分增加链接**：
   ```markdown
   详细案例参见 [docs/modes.md](../docs/modes.md#模式切换案例)
   ```

### 问题 6: 缺少"常见错误"速查表

**现象**：
- RULES.md 有"AI 常见违规行为速查"（第 243-256 行）
- 但只列出了违规行为，没有列出**用户的常见错误**
- 例如：用户说"继续"但 STATE.md 丢失怎么办？

**建议修复**：

1. **在 `docs/faq.md` 中增加"常见错误速查"章节**：
   ```markdown
   ## 常见错误速查
   
   ### 用户侧错误
   
   | 错误现象 | 原因 | 解决方法 |
   |----------|------|----------|
   | 说"继续"但 AI 无法恢复 | STATE.md 被删除或损坏 | 重新运行入场扫描，或手动指定从哪个阶段继续 |
   | 产物文件找不到 | 路径错误或命名不一致 | 检查 `.devflow-kit/<req-id>/` 目录，确认文件名符合规范 |
   | AI 跳过某个阶段 | 用户明确说"跳过"或模式为 Fast | 确认是否真的需要该阶段，如需要则回退重新执行 |
   | 模式选择后 AI 仍询问 | 未等待用户确认就继续提问 | 回复"已确认，请继续"或重新选择模式 |
   
   ### AI 侧错误
   
   | 错误现象 | 触发规则 | 正确做法 |
   |----------|----------|----------|
   | 未输出入场检测结果框 | R1.11 / R2.0 | 停止当前操作，先输出检测结果框 |
   | 产物遗漏模板段落 | R13.9 | 重新读取模板，补齐缺失段落后再生成 |
   | STATE.md 未更新 | R8.4 | 立即更新 STATE.md，再输出完成声明 |
   | 自行决定模式降级 | R10.3 | 暂停并询问用户，不得自行决定 |
   ```

### 问题 7: 缺少"多需求并行"的最佳实践

**现象**：
- R13.7 提到"多需求并行时使用 `.devflow-kit/<id>/` 隔离"
- 但没有说明如何管理多个 req-id
- 如何在不同需求间切换？

**建议修复**：

1. **在 `docs/workflow.md` 中增加"多需求并行"章节**：
   ```markdown
   ## 多需求并行最佳实践
   
   ### 场景：同时开发多个功能
   
   **推荐做法**：
   1. 为每个需求分配独立的 req-id：
      - `user-login`（用户登录）
      - `payment-integration`（支付集成）
      - `notification-system`（通知系统）
   
   2. 产物目录结构：
      ```
      .devflow-kit/
      ├── STATE.md                    # 全局状态（记录当前活跃 req）
      ├── CONTEXT.md                  # 共享上下文
      ├── user-login/                 # 需求 1
      │   ├── 00-requirements.md
      │   ├── 01-analysis.md
      │   └── ...
      ├── payment-integration/        # 需求 2
      │   ├── 00-requirements.md
      │   └── ...
      └── notification-system/        # 需求 3
          └── ...
      ```
   
   3. 切换需求：
      ```
      Use devflow-kit. 切换到 user-login
      ```
      AI 应该：
      - 更新 STATE.md 的 `活跃 req` 字段
      - 加载 `user-login/` 下的产物
      - 从上次中断的阶段继续
   
   4. 注意事项：
      - **不要同时激活多个 req**：STATE.md 的 `活跃 req` 只能有一个
      - **完成一个再开始下一个**：避免上下文混乱
      - **归档旧需求**：完成后移动到 `.devflow-kit/archive/<req-id>/`
   ```
