# DevFlow Kit

DevFlow Kit 是结构化软件开发工作的统一入口。它帮助 Agent 在合适的重量级流程下完成需求澄清、设计、任务拆分、实现、验证、审查与发布准备。

## 首要规则

1. **先读 `flow/GO.md`**：它是唯一入口和路由事实源。
2. **风险驱动选择流程**：小改动走 Fast，常规功能走 Standard，高风险改动走 Strict。
3. **产物写入项目文件**：Standard / Strict 的阶段产物放在 `.specs/<req-id>/`；Fast 默认只保留简短记录和验证证据。
4. **没有证据不声称完成**：完成声明必须包含测试、构建、lint、手动检查或无法验证的明确原因。
5. **不要悄悄扩大范围**：越界改动、破坏性操作、高风险决策必须暂停并请求用户确认。

## 什么时候使用

适用于：

- 模糊想法或新功能
- 非平凡 bugfix
- 多文件实现
- UI / 交互改动
- 架构、数据、迁移、性能、安全相关工作
- 测试规划、代码审查、发布准备
- 任何可能跨会话推进的任务

不建议为以下任务跑完整流程：

- 拼写错误、文案微调
- 单行或极小范围修复
- 纯解释代码或回答技术问题
- 用户明确说“不要走流程”

这类任务可以直接处理，但仍要做最小有效验证。

## 三种模式

| 模式 | 适用场景 | 默认行为 |
|---|---|---|
| **Fast** | ≤2 文件、低风险、需求清楚、容易回滚 | 可直接执行：短任务说明 → 修改 → 窄验证 → 结果证据 |
| **Standard** | 常规功能、多文件、业务逻辑、UI、需要验收准则 | 需要确认：需求确认 → 需求分析 → 方案设计 → 任务拆分 → 开发 → 测试 → 审查 → 集成 |
| **Strict** | 鉴权、支付、PII、schema、公共 API、CI/CD、部署、不可逆或生产敏感改动 | 必须确认：Standard 全流程 + 安全 / 迁移 / 回滚 / 独立 review |

模式确认规则以 `flow/GO.md` 为准：Fast 可默认继续；Standard 建议确认；Strict 必须确认。

## 核心目录

- `flow/GO.md`：统一路由器和加载策略。
- `flow/RULES.md`：全局红线与状态纪律。
- `flow/prompts/*.md`：各阶段执行 prompt。
- `flow/templates/*.md`：阶段产物模板。
- `flow/reference/*.md`：按需查阅的参考资料。
- `agent-skills/skills/*/_SKILL.md`：专业工程 skill。
- `adapters/`：不同工具的轻量适配器。

## 状态与产物

Standard / Strict 默认维护：

- `.specs/项目状态.md`：跨会话状态、模式、阶段、任务、入场扫描状态。
- `.specs/<req-id>/00-需求确认.md`
- `.specs/<req-id>/01-需求分析.md`
- `.specs/<req-id>/02-方案设计.md`
- `.specs/<req-id>/02a-UI设计.md`（仅前端 / UI）
- `.specs/<req-id>/03-任务拆分.md`
- `.specs/<req-id>/04-开发记录.md` 与任务级开发记录
- `.specs/<req-id>/05-测试报告.md`
- `.specs/<req-id>/06-代码审查.md`
- `.specs/<req-id>/07-发布清单.md`

本项目继续使用中文产物文件名；不要再要求英文文件名。内部状态枚举可使用英文，展示文档保持中文清晰可读。

## 阶段门

必须得到用户明确确认的事项：

- 需求范围最终确认
- 技术栈或架构方向锁定
- 新依赖、schema / migration、删除文件
- CI/CD、部署、生产发布、不可逆操作
- 接受 Critical review 发现而不修复
- 模式升级 / 降级

## 执行纪律

- 优先读当前文件和项目事实，不凭记忆假设。
- 长参考资料只读相关章节，不全文加载。
- 任务必须声明读取边界、写入边界、动作、验证方式和完成标准。
- 默认不创建 git commit；只有用户明确要求提交时才 commit。
- 遇到阻塞时说明证据、影响和最小决策点。

## 开始

读取 `flow/GO.md`，按它完成路由、模式判定、状态检查和阶段执行。
