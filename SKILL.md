# DevFlow

## ⛔ 入场前置检查（必须先读 GO.md）

**在继续之前，逐项确认：**

- [ ] 已完整读过 `flow/GO.md`（统一路由器）——不是跳过，不是只读摘要
- [ ] 已按 GO.md 第一步读取或创建项目状态（`.specs/进度跟踪.md`）
- [ ] 已按 GO.md 第二步跑完老项目入场检测（brownfield 必跑，greenfield 可跳过）
- [ ] 已按 GO.md 第三步路由表匹配用户意图
- [ ] 已按 GO.md 第三步 C 节验证阶段门（非 0-confirm 路由必须检查前置产物）
- [ ] 已输出路由声明（GO.md 第五步）

**如果以上任何一项未完成，立即停止**，先读 `flow/GO.md` 并按步骤执行。

> ⚠️ **常见违规模式**
> - 只读本文件（SKILL.md）就动手 → 跳过了路由、模式判定、入场检测
> - 看到 `.specs/` 有现有产物就默认跳转到开发 → 跳过了阶段门和入场协议
> - 用「我知道流程了」跳过 GO.md → GO.md 的路由表和加载策略经常更新，旧知识不适用
>
> **正确的做法**：每次调用 devflow-kit 时都重新读 `flow/GO.md` 的前几步（至少读到第三步路由表）。

## 概览

DevFlow 是一个面向结构化软件开发工作的统一入口。它把项目级交付流程和专业工程 skill 组合起来：`flow/` 层负责判断当前阶段、维护产物和阶段门，`agent-skills/skills/` 层负责提供每个阶段的专业工程纪律。

当任务不只是一个很小的孤立修改时，应使用这个 skill。它的目标是减少猜测、让跨会话上下文可恢复，并要求在声称完成前提供验证证据。

## 包结构

- `flow/GO.md` 是统一路由器。用户给出“做这个”“继续”“review”“测试”“执行 T03”这类宽泛指令时，先读它。
- `flow/SYSTEM.md`、`flow/METHODOLOGY.md` 和 `flow/RULES.md` 定义项目常驻规则、阶段门、产物体系和上下文纪律。
- `flow/prompts/*.md` 包含各阶段 prompt：需求确认、ANALYSIS、DESIGN、TASK、DEV、TEST、REVIEW、INTEGRATION，以及健康检查、架构梳理、视觉换皮等横向流程。
- `flow/templates/*.md` 包含阶段产物模板。
- `flow/reference/*.md` 包含较大的参考资料。除非文件很短，否则只读取当前需要的章节。
- `agent-skills/skills/*/_SKILL.md` 包含从上游 agent-skills 项目整合来的专业工程 skill。
- `agent-skills/agents/*.md` 包含可选的代码审查、安全审计、测试工程专家角色。
- `adapters/` 包含给支持斜杠命令、workflow 或 rules 文件的工具使用的轻量适配器。

## 什么时候使用

适用于模糊想法、新功能、老项目改造、架构决策、UI 工作、非平凡 bugfix、多文件实现、测试规划、代码审查、迁移、性能优化、安全敏感改动、CI/CD 工作、上线准备，以及任何可能跨多个会话推进的任务。

不建议为拼写错误、单行修改、或需求和验证方式都已经非常明确的单函数修改跑完整流程。即便如此，也仍然要使用相关的局部质量检查并验证改动。

## Token 预算估计（路由声明时用）

路由声明中追加一行：
```
✅ 当前模式预估：Standard full chain ~240k - 530k tokens
```
不做精确预估。实测偏差可达 30-60%（本次 `llm-rate-limit-retry` 实测 ~160-210k，表中 240-530k 因跳过部分测试和安全轮次而偏高。目的不是准，是让用户一眼知道「这大概会用多少 token」，而非做决策挡位用）。

## 模式选择（必须停等用户确认）

AI 判定模式后，**必须单独输出模式确认并停下来等用户回复**，不能和路由声明混在一起一次性输出。用户确认模式后 AI 才能输出路由声明并开始执行。

| 模式 | 适用场景 | 最小流程 |
|------|----------|----------|
| **Fast** | ≤2 文件、<50 行、低风险、需求清楚、可快速回滚 | 短任务说明 → 修改 → 窄验证 → 结果证据 |
| **Standard** | 常规功能、多文件、UI、业务逻辑、需验收准则 | 需求确认 → 需求分析 → 方案设计 → 任务拆分 → 开发 → 测试 → 审查 → 集成 |
| **Strict** | 高风险 / 生产敏感 / 架构或数据影响 | 完整流程 + 安全/迁移/回滚/独立 review |

格式和停等规则详见 `flow/GO.md`「模式确认」段。

## 模式降级（阶段间评估，需用户确认）

AI 在完成一个阶段进入下一阶段前，根据实际开发量评估是否应降级，但必须先询问用户确认：

| 信号 | 当前→目标 | 裁剪内容 |
|------|-----------|----------|
| 实际改动 ≤ 2 文件，< 50 行，且纯后/纯前端 | Standard → Fast | 跳过 2-design / 2a / 产物精简 |
| 用户表示"太多了" | 任何模式降一级 | — |

降级前必须暂停并询问用户确认（"📉 建议模式降级：Standard → Fast，理由：... 是否同意？[Y/n]"），用户同意后才执行。
Fast 模式内如发现隐藏的多模块影响才允许**升级**到 Standard，同样必须询问用户确认。

## 自动推进（阶段自动继续）

阶段 prompt 末尾的「触发下一步」遵循 R5.1（RULES.md）的自动继续规则：

- **默认行为**：完成当前阶段产物后，AI 直接进入下一阶段，不等"继续"指令
- **[Y/n] 规则**：TASK → DEV → TEST → REVIEW → INTEGRATION 之间的阶段门槛，AI 输出当前阶段产物后加一行 `✅ <阶段> 完成。继续？[Y/n]`。用户不回复 / 回复 Y → 继续；回复 n → 暂停
- **中断条件**：遇到 Critical 修复、模式需升级、多路由方向、需人工确认的门 → 必须停下来等用户

## 第一步

> 入场前置检查（见页首⛔入场前置检查）是必须的。如果你还没做，先做再继续。
> 本文件只描述模式和规则概览，**所有路由、入场检测、产物规范、自检清单都在 `flow/GO.md` 中**，
> 必须完整读取 `flow/GO.md` 后才能开始工作。

### 模式

Fast 模式用于低风险修改：通常只触碰 1~2 个文件、少于 50 行、需求清楚、不会影响公共契约。使用方式是写一个简短任务说明，直接修改，运行最窄但有意义的验证，并总结证据。除非用户要求，否则不要创建完整的 `.specs/<req-id>/` 生命周期。

Standard 模式用于常规功能、非平凡 bugfix、多文件实现、UI 改动，或任何预计需要超过一个专注会话完成的工作。使用 `flow/GO.md`，走 需求确认 → ANALYSIS → DESIGN → TASK → DEV → TEST → REVIEW 路径。**每个阶段必须产生对应的产物文件**（00-需求确认.md → 01-需求分析.md → 02-方案设计.md → 03-任务拆分.md → 开发总结 → 05-测试报告.md → 06-代码审查.md），不能跳过不写。只有用户明确说"不需要产物"时才可裁剪。

Strict 模式用于高风险或生产敏感改动。必须跑完整流程，在关键阶段要求明确确认，并按需要加入安全、性能、迁移、回滚检查；如果宿主环境支持独立专家角色，应使用独立 review。

### 风险矩阵

如果改动涉及以下任一项，升级到 Strict 模式：鉴权、授权、支付、账单、secret、PII、数据保留、数据库 schema 或 migration、公共 API、SDK、权限、基础设施、部署、CI/CD、环境配置、依赖升级、跨模块架构、并发、后台任务、缓存、数据丢失风险、不可逆操作、对回滚敏感的发布路径。

如果改动涉及多文件、用户可见 UI、业务逻辑、非平凡测试、外部集成、性能敏感代码，或需要验收准则的行为变化，使用 Standard 模式。

只有当改动局部、可逆、低风险且容易验证时，才使用 Fast 模式。

如果不确定，选择更重的模式，或让用户在 Fast、Standard、Strict 之间选择。

## 路由

所有 flow 驱动的工作必须按以下步骤执行：

1. **先读 `flow/GO.md`**，按上面的步骤完整执行（模式判定 → 意图解析 → 入场检测 → 加载 skill → 路由声明 → 执行 prompt）
2. **GO.md 中的规则覆盖本文件中的概览描述**。如果 Go.md 的要求与本文件冲突，以 GO.md 为准
3. **跳过的后果**：不读 GO.md 直接动手 → 不生成产物 → 不会做入场扫描 → 不会触发安全检查 → 用户拿到的是无验证、无记录的代码

**如果用户的要求很模糊，路由到 `flow/prompts/0-confirm.md`。如果已经明确是需求/设计/任务/开发/测试/审查，路由到对应的阶段 prompt。**

> 具体路由表见 `flow/GO.md` 的「第三步 · 路由表」。

## 阶段门

不要悄悄跳过阶段门。完整 flow 中，没有需求确认书就不要进入需求；没有需求就不要进入设计；没有包含验证方式的任务就不要写代码。如果用户明确要求跳过阶段，要说明取舍，并至少保留一个最小的书面事实源。

以下情况必须得到用户明确确认：最终确定需求范围、锁定技术栈或架构方向、修改数据库 schema、添加依赖、删除文件、执行迁移、修改 CI/CD 或部署配置、接受 Critical review 发现、归档或移动 flow 产物、宣布生产发布 GO。

没有证据就不要声称完成。验证证据可以是测试输出、构建输出、lint 输出、手动浏览器检查、截图、review 报告，或清楚说明为什么无法运行验证。

不要悄悄扩大范围。如果实现过程中发现新需求、架构问题或无关清理项，记录为 follow-up，或开一个新的 req，不要混进当前任务。

## 任务与并行规则

任务必须包含读取边界、写入边界、动作、可执行验证方式和完成标准。实现必须留在任务的写入边界内，除非用户批准更新任务。

并行任务是可选优化。只有当写入边界不重叠、依赖不冲突、共享状态或迁移风险可控时，才把任务标为可并行。禁止为了满足模板而硬造并行任务。

## 上下文纪律

优先相信文件和当前观察到的项目状态，而不是记忆或假设。引用历史决策时，指向对应产物文件。对较长参考资料，只 grep 或读取相关章节。不要因为包里有很多文件就全部加载。

除非宿主工具或项目另有明确约定，flow 产物应放在 `.specs/<req-id>/` 下。跨会话工作要维护 `进度跟踪.md`。

## 输出纪律

生成产物时，把它们写进项目文件，而不是只留在聊天里。对于没有文件写入能力的工具，提供带文件名的可复制产物块。

遇到阻塞时，说明阻塞条件、已经收集到的证据，以及需要用户做出的最小决策。不要通过编造缺失需求来继续推进。

## 验证清单

结束 flow 驱动任务之前，确认：已经遵循路由到的阶段 prompt；适用的专业 skill 已使用或有意跳过；必要产物已创建或更新；存在验证证据；剩余工作被记录为 pending，而不是暗示已经完成。
