## 1. AI不按流程走怎么办

### 症状

```
你: Use devflow-kit. 我想加个搜索功能。

AI: 好的，我来帮你实现搜索功能。首先...（直接开始写代码）
```

**问题**: AI跳过了需求确认、分析、设计等阶段，直接进入开发。

---

### 原因分析

1. **AI忘记加载GO.md** - 最常见
2. **用户未明确触发** - 只说"帮我做X"，没说"Use devflow-kit"
3. **上下文过长** - AI忘记了之前的指令

---

### 解决方法

#### 方法1: 重新触发（推荐）

```
@devflow-kit/flow/GO.md

我想加个搜索功能。
```

**关键点**: 
- 必须包含 `@devflow-kit/flow/GO.md`
- 或在Claude Code中说 "Use devflow-kit"

---

#### 方法2: 提醒AI遵守流程

```
请按照devflow-kit的流程执行：
1. 先进行需求确认
2. 然后需求分析
3. 再方案设计
...

不要直接写代码。
```

---

#### 方法3: 检查路由声明

AI应该输出类似这样的路由声明：

```
✅ 模式：Standard
✅ 路由：0-confirm
✅ Req-ID：add-search
✅ 已加载的 skill：
   - idea-refine
   - development-core
✅ 第一动作：澄清需求范围
```

**如果没有看到路由声明** → AI未按流程执行，需要重新触发。

---

### 预防措施

1. **每次新会话都重新触发** - 不要依赖AI记住
2. **检查第一条回复** - 确认有路由声明
3. **发现偏离立即纠正** - 不要等到后面才发现

## 2. 产物丢失如何处理

### 症状

```
你: 继续下一阶段

AI: 找不到 .specs/add-search/01-需求分析.md
```

---

### 原因分析

1. **文件被误删** - 用户或IDE清理
2. **路径错误** - AI读取了错误的路径
3. **Git操作** - checkout到其他分支

---

### 解决方法

#### 方法1: 检查文件是否存在

```bash
# 查看.specs/目录结构
ls -la .specs/add-search/

# 或 Windows
dir .specs\add-search\
```

**如果文件存在** → AI路径错误，告诉它正确路径：

```
文件在 .specs/add-search/01-需求分析.md，请重新读取。
```

**如果文件不存在** → 确实丢失，需要重建。

---

#### 方法2: 从Git恢复

```bash
# 查看文件历史
git log -- .specs/add-search/01-需求分析.md

# 恢复到最新版本
git checkout HEAD -- .specs/add-search/01-需求分析.md
```

---

#### 方法3: 让AI重建

```
01-需求分析.md 丢失了，请根据之前的对话重新生成。

提示：
- 我们之前讨论了搜索功能的用户故事
- 验收准则是：支持关键词搜索、按价格排序
- v1范围：仅商品名称搜索
```

**AI会基于记忆重建文件**。

---

### 预防措施

1. **定期提交Git** - 每次完成一个阶段就commit
2. **不要手动删除.specs/** - 除非归档
3. **使用.gitignore谨慎** - 确保不忽略重要产物

```gitignore
# 建议的.gitignore配置
.specs/archive/          # 归档的可以忽略
.specs/.memory/session-journal/  # 日志可以忽略
.specs/*.md              # ❌ 不要忽略主文件
.specs/*/                # ❌ 不要忽略req目录
```

## 3. 模式判定错误如何纠正

### 症状

```
AI: 🎯 模式推荐：Fast

你: 这个改动比想象中大，需要改5个文件
```

**问题**: AI低估了复杂度。

---

### 解决方法

#### 方法1: 显式指定模式

```
我选择Standard模式，因为：
1. 需要修改5个文件
2. 涉及API变更
3. 需要编写测试
```

**AI会立即切换到Standard流程**。

---

#### 方法2: 要求重新评估

```
请重新评估模式。实际情况：
- 改动文件数：5个（不是1-2个）
- 预估代码行数：150行（不是<50行）
- 涉及API接口变更

基于以上，应该用Standard还是Strict？
```

---

#### 方法3: 中途升级

如果在Fast模式执行中发现复杂度高：

```
⚠️ 我发现这个改动比预期复杂：
- 实际需要改5个文件
- 涉及数据库查询优化

建议升级到Standard模式，可以吗？
```

**AI会暂停当前工作，切换到Standard流程**。

---

### 模式升级规则

| 触发条件 | 建议操作 |
|---------|---------|
| 改动文件 > 2 | Fast → Standard |
| 代码行数 > 50 | Fast → Standard |
| 涉及支付/鉴权 | Standard → Strict |
| 需要数据迁移 | Standard → Strict |
| 用户明确要求 | 按需升级 |

## 4. 如何手动干预流程

### 场景1: 跳过某个阶段

```
你: 跳过需求分析，直接到设计阶段。

AI: ⚠️ 跳过警告

你要求跳过 1-analysis 阶段。

风险：
- 可能遗漏关键需求
- 验收标准不明确

确认跳过吗？
1. ✅ 确认跳过
2. 不跳过，补跑analysis
```

**AI会记录跳过原因，但允许你决定**。

---

### 场景2: 回到上一阶段

```
你: 我觉得设计有问题，回到设计阶段重新来。

AI: ✅ 回到 2-design

已重置：
- .specs/xxx/02-方案设计.md（标记为superseded）

请告诉我哪里不满意，我们重新设计。
```

---

### 场景3: 修改已有产物

```
你: 帮我修改 01-需求分析.md 的验收准则。

AI: ✅ 修改产物

原内容：
- AC1: 支持关键词搜索

新内容：
- AC1: 支持关键词模糊搜索
- AC2: 支持按价格/销量排序

已更新 .specs/xxx/01-需求分析.md
```

**可以直接让AI修改任何产物文件**。

---

### 场景4: 强制进入某阶段

```
你: 直接进入 4-dev 阶段，执行 T01。

AI: ⚠️ 阶段门检查

检测到缺失前置产物：
- 00-需求确认.md ✗
- 01-需求分析.md ✗
- 02-方案设计.md ✗
- 03-任务拆分.md ✗

选项：
1. 补跑缺失阶段（推荐）
2. 确认跳过所有前置阶段
3. 取消操作
```

**AI会阻止跳级，除非你明确确认**。

## 5. 常见错误消息解读

### 错误1: "未加载skill"

```
❌ 错误：进入 2-design 前必须加载 design-and-architecture skill
```

**原因**: AI忘记加载必需的skill  
**解决**: 提醒AI重新执行第五步的路由声明

---

### 错误2: "前置产物缺失"

```
❌ 错误：缺少 01-需求分析.md，无法进入 2-design
```

**原因**: 跳过了前一阶段或文件丢失  
**解决**: 补跑缺失阶段或重建文件

---

### 错误3: "Token超限"

```
⚠️ 警告：reference总行数200行，超过150行限制
```

**原因**: AI一次性读了太多参考文档  
**解决**: 让AI按节读取，不要整读

---

### 错误4: "模板校验失败"

```
❌ 错误：产物缺少必填段落「技术选型」
```

**原因**: AI未按模板输出  
**解决**: 让AI重新读取模板并补齐缺失段落

---

### 错误5: "模式未确认"

```
⚠️ 等待确认：Standard模式需要用户确认
```

**原因**: Standard/Strict模式必须等用户确认  
**解决**: 回复"确认"或选择其他模式

## 7. 最佳实践总结

### DO ✅

1. **每次新会话重新触发** - `@devflow-kit/flow/GO.md`
2. **检查路由声明** - 确认AI按流程执行
3. **定期提交Git** - 每个阶段完成后commit
4. **及时纠正偏差** - 发现不对立即指出
5. **善用手动干预** - 可以跳过、回退、修改

---

### DON'T ❌

1. **不要假设AI记得** - 每次都明确触发
2. **不要手动删除产物** - 除非归档
3. **不要忽略警告** - AI的警告通常很重要
4. **不要跳级太频繁** - 可能导致质量问题
5. **不要修改模板** - 保持标准格式
