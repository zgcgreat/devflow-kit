## 📋 问题清单与修复方案

### 问题 1: 任务拆分后没有详细实施计划

**现象**：
- 用户在 stage-3-task 完成后，直接进入开发阶段
- 缺少详细的时间规划、资源分配、风险预案

**根本原因**：
- stage-3-task 的 Step 8 询问逻辑不够明确
- Standard/Strict 模式下应该推荐生成实施计划，但提示文案不够清晰

**修复方案**：
✅ **已修复** - 修改 `flow/stage-skills/stage-3-task/_SKILL.md`

**修改内容**：
```markdown
### Step 8: 询问是否生成实施计划

**Standard/Strict 模式强制要求**：
```markdown
✅ 任务拆分完成，共X个任务。

📋 **下一步选择**：
1. 📝 生成详细实施计划（03a-实施计划.md）- **Standard/Strict 推荐**
   - 包含时间规划、资源分配、风险预案
   - 适合复杂项目、多任务并行场景
2. ⚡ 直接进入开发（4-dev）- **Fast 模式或简单项目**
   - 跳过实施计划，快速开始编码

请回复 1 或 2。
```

**Fast 模式**：自动进入 4-dev（不询问）
```

**效果**：
- Standard/Strict 模式下会明确推荐生成实施计划
- 用户可以看到两个选项的详细说明
- Fast 模式保持简洁，自动跳过

### 问题 2: 任务实施时不用每个小任务都确认，且有些任务可以并行执行

**现象**：
- 每完成一个任务都要等待用户确认才能继续
- 并行任务（[P]标记）也没有充分利用

**根本原因**：
- stage-4-dev 的触发下一步逻辑过于保守
- 没有区分顺序任务和并行任务的处理方式

**修复方案**：
✅ **已修复** - 修改 `flow/stage-skills/stage-4-dev/_SKILL.md`

**修改内容**：
```markdown
### Step 12: 路由到下一个任务或下一阶段

**检查是否还有未执行的任务**：

1. 读取 `03-tasks.md`，检查所有任务的 `status` 字段
2. 如果存在 `status="pending"` 的任务：
   - **如果是 [P] 标记的并行任务**：
     ```markdown
     ✅ T01 完成。
     
     🔄 **检测到可并行任务**：
     - T02[P]（与T01无依赖冲突）
     - T03[P]（与T01无依赖冲突）
     
     💡 **建议**：可以立即执行 T02 或 T03，无需等待确认。
     
     请指定下一个任务ID（如 "T02"），或说 "继续" 自动选择下一个。
     ```
   - **如果是顺序任务**：
     ```markdown
     ✅ T01 完成。
     
     ➡️ **下一步**：T02（依赖 T01，现已满足）
     
     正在加载 T02 任务定义...
     ```
     → 自动加载下一个任务，无需用户确认
   
3. 如果所有任务都已完成：
   ```markdown
   ✅ 所有任务完成！
   
   📊 **任务统计**：
   - 总任务数：X
   - 已完成：X
   - 并行任务：Y个
   
   ➡️ **进入测试阶段**（stage-5-test）
   ```
   → 加载 `flow/stage-skills/stage-5-test/_SKILL.md`
```

**效果**：
- 顺序任务自动继续，无需用户确认
- 并行任务给出建议，用户可以指定或说"继续"
- 所有任务完成后自动进入测试阶段

### 问题 3: 04-dev-log.md + 任务级记录这个文档显示已生成，实际没有

**现象**：
- STATE.md 显示 `04-dev-log.md + 任务级记录` 已生成
- 但实际文件系统中找不到对应的文件

**根本原因**：
- 文件名命名不规范，模板叫 `04-dev-log.md`，但实际产物应该按任务ID命名
- Stage Skill 中的注释和实际输出不一致

**修复方案**：
✅ **已修复** - 修改 `flow/stage-skills/stage-4-dev/_SKILL.md`

**修改内容**：
```markdown
## 输出

- 代码文件（按任务的write_files）
- `.devflow-kit/<req-id>/<task-id>-dev-log.md`（如 `T01-dev-log.md`）

> **注意**: 模板文件名为 `04-dev-log.md`,但实际产物会根据task id命名为 `<task-id>-dev-log.md` (如 `T01-dev-log.md`)。每个任务都有独立的开发记录。
```

**效果**：
- 明确了实际产物命名规则：`<task-id>-dev-log.md`
- 每个任务都有独立的开发记录文件
- STATE.md 的描述需要后续更新为"任务级开发记录（T01-dev-log.md, T02-dev-log.md, ...）"

### 问题 4: 是否每个阶段都正确加载了子skill，在进入每个阶段时都描述下本阶段要使用什么skill

**现象**：
- AI 加载 Stage Skill 时没有明确说明
- 用户不知道当前阶段使用了哪些 skill 和依赖

**根本原因**：
- GO.md Step 4 只要求加载 skill，但没有要求输出加载声明

**修复方案**：
✅ **已修复** - 修改 `flow/GO.md`

**修改内容**：
```markdown
## Step 4 · 加载Stage Skill

**加载规则**：
1. 根据 Step 3 路由表确定目标 Stage
2. **输出加载声明**（必须包含以下信息）：
   ```markdown
   📦 正在加载 Stage Skill: <stage-name>
   - Skill路径: flow/stage-skills/<path>/_SKILL.md
   - 依赖Skills: <dependency-list 或 "无">
   - 本阶段目标: <简要描述>
   ```
3. 只加载该 Stage 的 _SKILL.md（不要加载其他 Stage）
4. 如声明 dependencies，先加载依赖 Skills
5. 如 Stage Skill 不可用，降级到 Prompt 文件
```

**效果示例**：
```markdown
📦 正在加载 Stage Skill: stage-0-confirm
- Skill路径: flow/stage-skills/stage-0-confirm/_SKILL.md
- 依赖Skills: idea-refine, development-core
- 本阶段目标: 澄清需求并判定执行模式
```

### 问题 5: 项目执行完后是否更新记忆

**现象**：
- 需求完成后，记忆系统没有自动更新
- 下次会话时 AI 不记得之前做过什么

**根本原因**：
- stage-7-integration 只有伪代码，没有实际的记忆更新逻辑
- 缺少具体的更新步骤和格式

**修复方案**：
✅ **已修复** - 修改 `flow/stage-skills/stage-7-integration/_SKILL.md`

**新增 Step 8: 更新记忆系统（强制执行）**：

#### 8.1 检查记忆系统是否存在
```python
if exists(".devflow-kit/memory/"):
    print("✅ 检测到记忆系统，开始更新...")
else:
    print("ℹ️ 未检测到记忆系统，跳过更新")
```

#### 8.2 更新 CURRENT_STATE.md
提取本次需求的关键信息，更新当前状态。

#### 8.3 追加 DECISIONS.md（如有新决策）
从 02-design.md 和 06-code-review.md 提取重要决策。

#### 8.4 追加 KNOWN_FAILURES.md（如有新失败模式）
从 05-test-report.md 和 06-code-review.md 提取失败模式。

#### 8.5 创建会话日志
在 `.devflow-kit/memory/journals/` 下创建 `YYYY-MM-DD-<req-id>.md`。

#### 8.6 输出更新摘要
```markdown
🧠 **记忆系统已更新**

✅ CURRENT_STATE.md - 已更新当前状态
✅ DECISIONS.md - 已追加 X 个新决策
✅ KNOWN_FAILURES.md - 已追加 Y 个新失败模式
✅ journals/YYYY-MM-DD-<req-id>.md - 已创建会话日志

💡 **下次会话时**，AI会自动读取这些记忆，无需重复沟通。
```

**效果**：
- 每个需求完成后自动更新记忆系统
- 记录关键决策、失败模式、会话日志
- 下次会话时 AI 会自动读取这些记忆

- 担心记忆系统只在第一次有效

**根本原因**：
- stage-0-confirm 没有明确说明记忆读取机制

**修复方案**：
✅ **已修复** - 修改 `flow/stage-skills/stage-0-confirm/_SKILL.md`

**修改内容**：
```markdown
#### 1.1 扫描主流程产物

| 产物文件 | 存在性 | 优先级 | 提取内容 |
|---------|--------|--------|----------|
| STATE.md | ✅/❌ | 🔴 必须 | 活跃req、中断任务、当前阶段 |
| CONTEXT.md | ✅/❌ | 🟡 建议 | 技术栈、编码规范、禁动清单 |
| **memory/PROJECT_CONTEXT.md** | ✅/❌ | 🟡 建议 | **项目背景、历史决策、已知失败** |
| lessons-learned.md | ✅/❌ | 🟢 可选 | 历史教训、最佳实践 |

**🟡 建议读取**（缺失采用降级策略）：
- **memory/PROJECT_CONTEXT.md**：提供项目历史和背景

  - 如果缺失 → 提示用户初始化记忆系统（`Use manage-memory`）

**效果**：
- 每个新需求开始时，AI 会自动读取记忆系统
- 包括 PROJECT_CONTEXT.md、DECISIONS.md、KNOWN_FAILURES.md
- AI 会记住项目的历史决策和已知失败，避免重复踩坑

**根本原因**：
- GO.md 没有明确说明多项目支持机制

**修复方案**：

**新增说明**：
```markdown
5. **多项目支持说明**：
   - DevFlow Kit **支持多项目并行**，每个项目有独立的 `.devflow-kit/` 目录
